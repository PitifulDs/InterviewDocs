# 面试基础题整理

## 2. TCP / UDP / IP

### IP 是什么

IP（Internet Protocol）是**网络层协议**，核心作用：

- 给数据包提供**地址**
- 负责把数据从一台机器送到另一台机器
- 本身是**无连接、不可靠**的，只负责尽力转发

可以理解为：

- **IP = 地址 + 路由转发**
- 不保证送达
- 不保证顺序
- 不保证不重复

常见版本：

- IPv4：32 位地址
- IPv6：128 位地址

### TCP 是什么

TCP（Transmission Control Protocol）是**传输层协议**，特点：

- **面向连接**
- **可靠传输**
- **有序**
- **有流量控制**
- **有拥塞控制**

TCP 为了保证可靠，做了很多事情：

#### 1）三次握手建立连接

目的是双方确认：

- 你能发给我
- 我能发给你
- 双方都知道初始序列号

流程：

- 客户端发 SYN
- 服务端回 SYN + ACK
- 客户端回 ACK

#### 2）四次挥手断开连接

因为 TCP 是全双工，双方都要单独关闭发送方向。

#### 3）可靠性的实现

TCP 通过这些机制实现可靠：

- 序列号 seq
- 确认应答 ack
- 超时重传
- 滑动窗口
- 流量控制
- 拥塞控制

#### 4）面向字节流

TCP 没有“消息边界”，应用层自己做拆包/封包，所以会有：

- 粘包
- 拆包

常见解决方法：

- 固定长度
- 分隔符
- 长度字段 + body

### UDP 是什么

UDP（User Datagram Protocol）也是传输层协议，但它：

- **无连接**
- **不保证可靠**
- **不保证有序**
- **报文边界清晰**
- 开销小
- 速度快
- 实时性好

特点：

- 不需要握手
- 发出去就发出去
- 不保证对方收到
- 适合实时场景

常见应用：

- 音视频
- DNS
- 广播/组播
- 游戏实时同步

### TCP 和 UDP 的区别

#### TCP

- 面向连接
- 可靠
- 有序
- 面向字节流
- 开销大
- 适合文件传输、HTTP、数据库通信

#### UDP

- 无连接
- 不可靠
- 无序
- 面向报文
- 开销小
- 适合实时通信、直播、语音、DNS

### 面试总结

> IP 负责寻址和路由转发，TCP/UDP 位于传输层。TCP 提供可靠、有序、面向连接的字节流服务；UDP 提供无连接、低开销、面向报文的传输服务。选择 TCP 还是 UDP，本质上是在**可靠性**和**实时性/性能**之间做权衡。

## 3. IO 多路复用介绍

IO 多路复用的核心思想：

> **一个线程同时监听多个文件描述符，当某些 fd 就绪后再去处理。**

好处：

- 不需要一个连接一个线程
- 减少线程切换开销
- 提高高并发能力

### 常见模型

#### 1）select

特点：

- 通过 `fd_set` 传入要监听的 fd 集合
- 内核返回后告诉你哪些 fd 就绪了

缺点：

- 有最大 fd 数量限制
- 每次都要把整个集合从用户态拷贝到内核态
- 返回后还要遍历整个集合
- 效率一般

#### 2）poll

和 select 类似，但用 `pollfd` 数组表示 fd，解决了 `select` 的位图大小限制。

缺点：

- 仍然要遍历所有 fd
- 性能提升有限

#### 3）epoll（Linux 高频）

Linux 下最常用。

核心优势：

- 把“注册 fd”和“等待事件”分开
- 内核维护就绪链表
- 用户态不需要每次遍历所有 fd
- 适合大量连接场景

核心接口：

- `epoll_create`
- `epoll_ctl`
- `epoll_wait`

### epoll 为什么高效

因为它把 select/poll 的两个问题优化了：

#### 1）不用每次重复传所有 fd

fd 注册一次即可，之后由内核维护。

#### 2）不用遍历全部 fd

只返回“就绪的 fd”。

所以大量连接下性能更好。

### LT 和 ET

#### LT（水平触发）

只要 fd 里还有数据没读完，就会一直通知。

优点：

- 简单
- 不容易丢事件

#### ET（边缘触发）

只有状态变化时通知一次。

优点：

- 通知次数少，效率高

缺点：

- 必须一次性把数据尽量读完，通常要配合**非阻塞 IO**
- 写代码更难

### 面试总结

> IO 多路复用就是让一个线程监听多个 fd，避免“一连接一线程”的高成本模型。select/poll 本质上都需要遍历所有 fd，而 epoll 通过事件驱动和就绪链表避免了无效遍历，因此更适合高并发网络服务器。

## 4. 程序到可执行文件的处理过程

标准流程：

### 1）预处理

处理：

- `#include`
- `#define`
- 条件编译

输出：**预处理后的源文件**

```bash
g++ -E main.cpp -o main.i
```

### 2）编译

把 C/C++ 源码转成**汇编代码**。

做的事情：

- 词法分析
- 语法分析
- 语义分析
- 中间代码优化
- 生成汇编

```bash
g++ -S main.i -o main.s
```

### 3）汇编

把汇编代码转成**目标文件 `.o`**。

```bash
g++ -c main.s -o main.o
```

目标文件里通常有：

- 机器指令
- 符号表
- 重定位信息
- 数据段信息

### 4）链接

把多个 `.o` 和库文件链接成最终可执行文件。

链接器主要做：

- 符号解析
- 地址重定位
- 合并代码段/数据段

```bash
g++ main.o -o app
```

### 链接时干了什么

比如在 `main.cpp` 调用了 `add()`，但实现写在 `add.cpp`。

- 编译阶段只知道“有这么个符号”
- 真正把它们拼起来是在链接阶段完成的

### 动态链接和静态链接

链接时可能：

- 把库代码直接拷进可执行文件 —— **静态链接**
- 只保留对共享库的引用，运行时再加载 —— **动态链接**

### 面试回答模板

> C++ 程序从源码到可执行文件一般经历四步：预处理、编译、汇编、链接。预处理展开头文件和宏；编译把源码转成汇编；汇编生成目标文件；链接阶段完成符号解析和地址重定位，最终形成可执行文件。

## 5. 动态库和静态库的区别

### 静态库

Linux 下一般是 `.a`

特点：

- 链接阶段把库代码直接拷贝到可执行文件
- 生成的可执行文件更大
- 运行时不依赖外部库文件
- 部署简单

### 动态库

Linux 下一般是 `.so`

特点：

- 链接时只记录引用
- 运行时由动态链接器加载
- 可执行文件更小
- 多个进程可共享库代码段
- 更新库时有时不需要重新编译主程序
- 但部署时要保证 `.so` 存在且版本兼容

### 区别总结

#### 静态库

优点：

- 部署简单
- 运行时依赖少

缺点：

- 可执行文件大
- 更新库要重新链接
- 多进程无法共享同一份库代码副本

#### 动态库

优点：

- 文件小
- 节省内存
- 易于升级

缺点：

- 部署复杂
- 存在版本兼容问题
- 启动时需要动态装载

### 面试一句话

> 静态库是在链接阶段把库代码拷进可执行文件，动态库是在运行时加载共享库。静态库部署方便但体积大，动态库节省空间但依赖运行环境。

## 6. 内存对齐

### 什么是内存对齐

编译器会按一定规则给变量分配地址，让数据地址满足某种倍数要求，这叫**对齐**。

例如：

- `int` 通常要求 4 字节对齐
- `double` 通常要求 8 字节对齐

### 为什么要对齐

主要两个原因：

#### 1）CPU 访问效率更高

CPU 通常按字长读取数据，对齐后可以一次取完。

#### 2）某些架构不对齐访问会出错

有些 CPU 对未对齐访问非常敏感。

### 结构体对齐规则

常见规则：

1. 成员按照各自对齐要求放置
2. 当前偏移必须是该成员对齐数的整数倍
3. 整个结构体大小必须是“最大对齐数”的整数倍

### 例子

```cpp
struct A {
    char c;
    int i;
};
```

分析：

- `c` 占 1 字节
- 为了让 `i` 4 字节对齐，要补 3 字节
- `i` 占 4 字节

总共：8 字节

再看：

```cpp
struct B {
    char c1;
    char c2;
    int i;
};
```

分析：

- `c1` 1
- `c2` 1
- 补 2
- `i` 4

总共：8 字节

### 如何减少内存浪费

把大对齐成员放前面，小的放后面，通常可以减小结构体大小。

### `#pragma pack`

可以修改对齐规则，但：

- 可能降低性能
- 甚至导致某些平台访问异常

一般只在以下场景使用：

- 网络协议
- 二进制文件格式
- 与硬件/协议严格对接

### 面试回答

> 内存对齐是编译器为了提高访问效率并满足硬件要求，对变量地址按一定边界进行对齐。结构体大小不仅取决于成员大小，还取决于成员排列顺序和最大对齐要求。

## 7. 加入虚函数多少字节

### 结论

类里只要有虚函数，通常对象里会多一个虚函数表指针 `vptr`。

- **64 位系统下**：`vptr` 一般是 **8 字节**
- **32 位系统下**：`vptr` 一般是 **4 字节**

### 例子

```cpp
class A {
public:
    virtual void func() {}
};
```

此时 `A` 对象里通常会有一个 `vptr`。

### 为什么不能死记

因为 `sizeof(A)` 还受以下因素影响：

- 编译器实现
- ABI
- 对齐规则

例如：

```cpp
class A {
public:
    virtual void func() {}
    char c;
};
```

在 64 位下：

- `vptr`：8
- `c`：1
- 补齐到 8 的倍数

所以通常 `sizeof(A) = 16`

### 面试稳妥说法

> 类中加入虚函数后，对象中通常会增加一个虚表指针 vptr。64 位系统下一般增加 8 字节，32 位下一般增加 4 字节，但最终 `sizeof` 还要看成员和对齐规则。

## 8. 派生类继承基类有虚函数，占多少字节

### 情况 1：基类有虚函数，子类没有新增虚函数

```cpp
class Base {
public:
    virtual void f() {}
};

class Derive : public Base {
    int x;
};
```

分析：

- `Base` 已经有虚函数了，所以对象里有一个 `vptr`
- `Derive` 继承 `Base` 后，**子类对象通常仍然只有一个 `vptr`**
- 不会因为“继承了虚函数”再多一个 `vptr`

在 64 位常见情况下：

- `Base` 通常 8
- `int x` 4
- 补齐 4

所以 `sizeof(Derive)` 常见是 **16**

### 情况 2：子类重写虚函数

不会额外多一个 `vptr`，只是虚表内容变了。

### 情况 3：子类新增虚函数

通常仍然只有一个 `vptr`，只是虚表会扩展。

### 面试回答

> 如果基类已经有虚函数，那么派生类对象通常也只保留一个 `vptr`，不会因为继承或重写虚函数再额外增加一个。最终大小等于基类子对象大小加上派生类成员，再考虑内存对齐。

## 9. 两个类中类成员有另一个类的变量，再加上虚函数，占多少字节

这题考的是：**对象组合 + 虚函数 + 对齐**。

### 例子 1

```cpp
class A {
public:
    virtual void f() {}
    int x;
};

class B {
public:
    char c;
    A a;
};
```

先算 `A`：

- `vptr`：8
- `int`：4
- 补齐：4

所以 `sizeof(A) = 16`

再算 `B`：

- `char c`：1
- 为了让 `A a` 按 8 对齐，补 7
- `A a`：16

总共：24

所以 `sizeof(B) = 24`

### 例子 2

```cpp
class A {
public:
    virtual void f() {}
};

class B {
public:
    A a;
    int x;
};
```

分析：

- `A` 一般 8
- `int x` 4
- 补 4

所以 `sizeof(B)` 常见是 **16**

### 规则总结

组合类大小计算规则：

1. 成员依次排布
2. 每个成员按自己的对齐要求对齐
3. 最终总大小按最大对齐数补齐

如果成员类里有虚函数：

- 那成员对象内部通常已经带有 `vptr`
- 所以它的 `sizeof` 已经包含这部分开销

### 面试说法

> 如果一个类里包含另一个类对象作为成员，那么成员对象不是指针，而是直接嵌入对象内存布局中，因此要把该成员类的完整 `sizeof` 算进去。如果成员类含有虚函数，其 `vptr` 开销已经体现在该成员类大小中，外层类再按对齐规则计算总大小。

## 10. 智能指针介绍

C++ 常见智能指针有三种：

- `unique_ptr`
- `shared_ptr`
- `weak_ptr`

目的：**自动管理资源生命周期，减少内存泄漏和悬空指针问题。**

### 1）`unique_ptr`

独占所有权，同一时刻只能有一个拥有者。

特点：

- 不能拷贝
- 可以移动
- 开销小
- 最符合 RAII 思想

例子：

```cpp
std::unique_ptr<int> p = std::make_unique<int>(10);
```

适合场景：

- 明确唯一拥有者
- 工厂函数返回对象
- 资源独占管理

### 2）`shared_ptr`

共享所有权，多个 `shared_ptr` 可以指向同一对象。

特点：

- 内部有**引用计数**
- 最后一个拥有者析构时释放资源

例子：

```cpp
std::shared_ptr<int> p1 = std::make_shared<int>(10);
std::shared_ptr<int> p2 = p1;
```

适合场景：

- 多处共同持有对象
- 对象生命周期不容易人工判断时

缺点：

- 有引用计数开销
- 可能出现循环引用

### 3）`weak_ptr`

弱引用，不增加引用计数，用来观察 `shared_ptr` 管理的对象。

作用：

- 打破循环引用
- 判断对象是否还活着

例子：

```cpp
std::weak_ptr<int> wp = p1;
if (auto sp = wp.lock()) {
    // 对象还活着
}
```

### 面试总结

> `unique_ptr` 表示独占所有权，轻量高效；`shared_ptr` 表示共享所有权，依赖引用计数；`weak_ptr` 是对 `shared_ptr` 的弱引用，不参与计数，主要用于解决循环引用问题。

## 11. `shared_ptr` 中的 `enable_shared_from_this` 是干什么的

### 它解决什么问题

如果一个对象已经被 `shared_ptr` 管理，你在对象内部想拿到“指向自己的 `shared_ptr`”，不能直接写：

```cpp
std::shared_ptr<MyClass>(this)
```

因为这样会**重新创建一个独立控制块**，导致：

- 一个对象被两个独立 `shared_ptr` 管理
- 最终重复释放
- 程序崩溃

### `enable_shared_from_this` 的作用

它允许对象内部安全地获得一个**和外部共享同一个控制块**的 `shared_ptr<this>`。

### 例子

```cpp
#include <memory>
#include <iostream>

class MyClass : public std::enable_shared_from_this<MyClass> {
public:
    std::shared_ptr<MyClass> getSelf() {
        return shared_from_this();
    }
};

int main() {
    auto p1 = std::make_shared<MyClass>();
    auto p2 = p1->getSelf();

    std::cout << p1.use_count() << std::endl; // 2
}
```

这里 `p2` 和 `p1` 共享同一个引用计数控制块。

### 注意事项

#### 1）对象必须已经被 `shared_ptr` 管理

如果对象不是由 `shared_ptr` 创建的，直接调用 `shared_from_this()` 会出异常。

错误示例：

```cpp
MyClass obj;
obj.shared_from_this(); // 错
```

#### 2）推荐配合 `make_shared`

这样最安全。

### 面试回答

> `enable_shared_from_this` 的作用是让对象内部安全地获得指向自己的 `shared_ptr`，并且与外部已有的 `shared_ptr` 共用同一个控制块，避免直接 `shared_ptr(this)` 导致重复释放的问题。

## 12. `unique_ptr` 可以作为函数的返回值吗

### 可以，而且非常推荐

因为 `unique_ptr` 不能拷贝，但**可以移动**。

### 例子

```cpp
#include <memory>

std::unique_ptr<int> createObj() {
    return std::make_unique<int>(100);
}

int main() {
    std::unique_ptr<int> p = createObj();
}
```

这里返回值会发生：

- RVO / NRVO
- 或移动构造

所以完全没问题。

### 为什么很常用

工厂函数特别适合返回 `unique_ptr`：

```cpp
class Base {
public:
    virtual ~Base() = default;
};

class Derived : public Base {};

std::unique_ptr<Base> create() {
    return std::make_unique<Derived>();
}
```

这在现代 C++ 里非常常见。

### 面试回答

> `unique_ptr` 可以作为函数返回值，因为它支持移动语义。返回时通常触发返回值优化或移动构造，因此这是一种非常标准、推荐的现代 C++ 写法，尤其适合工厂模式。

## 13. 互斥锁 + 条件变量

### 互斥锁 `mutex`

作用：

- 保证同一时刻只有一个线程访问共享资源
- 防止数据竞争

### 条件变量 `condition_variable`

作用：

- 让线程在某个条件不满足时**睡眠等待**
- 避免一直 `while` 轮询浪费 CPU

### 经典场景：生产者消费者

消费者发现队列为空：

- 不应该一直轮询
- 应该阻塞等待

生产者放入数据后：

- 通知消费者继续工作

### 基本用法

#### 等待方

```cpp
std::mutex mtx;
std::condition_variable cv;
std::queue<int> q;

void consumer() {
    std::unique_lock<std::mutex> lock(mtx);
    cv.wait(lock, [] { return !q.empty(); });
    int val = q.front();
    q.pop();
}
```

#### 通知方

```cpp
void producer() {
    {
        std::lock_guard<std::mutex> lock(mtx);
        q.push(10);
    }
    cv.notify_one();
}
```

### 为什么 `wait` 要传 `unique_lock`

因为 `wait` 的过程是：

1. 检查条件
2. 条件不满足时，**原子地释放锁并阻塞**
3. 被唤醒后，重新加锁
4. 再检查条件

所以需要一个可解锁再重锁的锁对象，`lock_guard` 做不到，`unique_lock` 可以。

### 为什么要用 lambda 判断条件

因为可能存在：

- **虚假唤醒**
- 多线程竞争导致条件再次不满足

所以必须写成：

```cpp
cv.wait(lock, [] { return 条件成立; });
```

而不是：

```cpp
if (!条件) cv.wait(lock);
```

### 面试回答

> 互斥锁用于保护共享资源，条件变量用于线程间同步，避免线程在条件不满足时忙等。`wait` 会在阻塞时释放锁，被唤醒后重新加锁，因此通常搭配 `unique_lock` 使用，并通过谓词防止虚假唤醒。

## 14. `future` 和 `packaged_task`

### `future` 是什么

`std::future` 用来获取**异步结果**。

可以理解为：

> “一个将来会拿到结果的对象”

### 常见获取 `future` 的方式

- `std::async`
- `std::promise`
- `std::packaged_task`

### `packaged_task` 是什么

`std::packaged_task` 本质上是：

> **把一个可调用对象包装起来，使它执行后结果能放进 `future` 里。**

### 例子

```cpp
#include <future>
#include <iostream>

int add(int a, int b) {
    return a + b;
}

int main() {
    std::packaged_task<int(int, int)> task(add);
    std::future<int> fut = task.get_future();

    std::thread t(std::move(task), 3, 4);
    t.join();

    std::cout << fut.get() << std::endl; // 7
}
```

### 过程解释

#### `packaged_task` 做了什么

- 包装函数 `add`
- 调用时执行函数
- 自动把结果存进关联的共享状态

#### `future` 做了什么

- 从共享状态里取结果
- `get()` 会阻塞直到结果准备好

### 和 `async` 的区别

#### `std::async`

更高级，帮你自动启动异步任务。

#### `std::packaged_task`

更底层，你自己决定：

- 什么时候执行
- 哪个线程执行
- 是否放到线程池中执行

所以在线程池设计里更常见。

### 面试回答

> `future` 用于获取异步任务结果，`packaged_task` 用于把一个可调用对象包装成可异步执行的任务，并把执行结果关联到一个 `future`。`future` 负责取结果，`packaged_task` 负责产出结果。

## 15. `memory_order`（内存序 / 内存屏障）

### 它解决什么问题

多线程下，编译器和 CPU 为了优化性能，可能会对指令进行**重排序**。

单线程下重排通常没问题；但多线程下，如果不控制顺序，就可能出现一个线程看到了“半初始化”的结果。

所以 C++ 原子操作提供了 `memory_order` 来约束可见性和顺序。

### 常见内存序

#### 1）`memory_order_relaxed`

只保证原子性，不保证顺序。

适合：

- 只关心计数值本身
- 不关心和别的读写顺序关系

```cpp
std::atomic<int> cnt{0};
cnt.fetch_add(1, std::memory_order_relaxed);
```

#### 2）`memory_order_acquire`

用于“读”的一侧。

含义：

- 当前读之后的操作，不能被重排到这个读之前
- 通常和 `release` 配对使用

#### 3）`memory_order_release`

用于“写”的一侧。

含义：

- 当前写之前的操作，不能被重排到这个写之后

#### 4）`memory_order_acq_rel`

同时具备 acquire 和 release 语义，常用于读改写操作。

#### 5）`memory_order_seq_cst`

最强语义：

- 保证全局看起来像一个统一顺序
- 最容易理解
- 性能可能略差

默认原子操作通常就是这个。

### acquire / release 经典理解

线程 A：

```cpp
data = 100;
flag.store(true, std::memory_order_release);
```

线程 B：

```cpp
if (flag.load(std::memory_order_acquire)) {
    std::cout << data << std::endl;
}
```

含义：

- A 线程在 `release` 前对 `data` 的写入
- 对 B 线程在看到 `flag = true` 后是可见的

这是典型的“发布-订阅”模型。

### 面试回答

> `memory_order` 用于约束原子操作的可见性和重排序行为。`relaxed` 只保证原子性，`acquire/release` 用于线程间同步建立 happens-before 关系，`seq_cst` 提供最强顺序一致性语义。实际工程中如果没有明确性能诉求，优先使用默认的 `seq_cst`，避免错用内存序。

## 手撕：字符串的四则运算

### 题目思路

输入一个字符串表达式，支持：

- `+ - * /`
- 空格
- 括号
- 正负号
- 整数运算

推荐写法：**递归下降解析**。

分层：

- 表达式层：处理 `+ -`
- 项层：处理 `* /`
- 因子层：处理数字、括号、一元正负号

这样天然符合数学优先级。

### C++ 实现

```cpp
#include <iostream>
#include <string>
#include <cctype>
#include <stdexcept>

class Calculator {
public:
    int calculate(const std::string& s) {
        int index = 0;
        return parseExpression(s, index);
    }

private:
    void skipSpaces(const std::string& s, int& i) {
        while (i < (int)s.size() && std::isspace((unsigned char)s[i])) {
            ++i;
        }
    }

    // expression = term { ('+' | '-') term }
    int parseExpression(const std::string& s, int& i) {
        int value = parseTerm(s, i);
        while (true) {
            skipSpaces(s, i);
            if (i >= (int)s.size()) break;

            char op = s[i];
            if (op != '+' && op != '-') break;
            ++i;

            int rhs = parseTerm(s, i);
            if (op == '+') value += rhs;
            else value -= rhs;
        }
        return value;
    }

    // term = factor { ('*' | '/') factor }
    int parseTerm(const std::string& s, int& i) {
        int value = parseFactor(s, i);
        while (true) {
            skipSpaces(s, i);
            if (i >= (int)s.size()) break;

            char op = s[i];
            if (op != '*' && op != '/') break;
            ++i;

            int rhs = parseFactor(s, i);
            if (op == '*') value *= rhs;
            else {
                if (rhs == 0) {
                    throw std::runtime_error("divide by zero");
                }
                value /= rhs;
            }
        }
        return value;
    }

    // factor = number | '(' expression ')' | ('+'|'-') factor
    int parseFactor(const std::string& s, int& i) {
        skipSpaces(s, i);

        if (i >= (int)s.size()) {
            throw std::runtime_error("invalid expression");
        }

        if (s[i] == '+') {
            ++i;
            return parseFactor(s, i);
        }
        if (s[i] == '-') {
            ++i;
            return -parseFactor(s, i);
        }

        if (s[i] == '(') {
            ++i;
            int value = parseExpression(s, i);

            skipSpaces(s, i);
            if (i >= (int)s.size() || s[i] != ')') {
                throw std::runtime_error("missing ')'");
            }
            ++i;
            return value;
        }

        if (std::isdigit((unsigned char)s[i])) {
            int value = 0;
            while (i < (int)s.size() && std::isdigit((unsigned char)s[i])) {
                value = value * 10 + (s[i] - '0');
                ++i;
            }
            return value;
        }

        throw std::runtime_error("invalid character");
    }
};

int main() {
    Calculator calc;

    std::string s1 = "1 + 2 * 3";
    std::string s2 = "(1 + 2) * 3";
    std::string s3 = "-2 + 3 * (4 - 1)";
    std::string s4 = "10 + 6 / 2";

    std::cout << s1 << " = " << calc.calculate(s1) << std::endl; // 7
    std::cout << s2 << " = " << calc.calculate(s2) << std::endl; // 9
    std::cout << s3 << " = " << calc.calculate(s3) << std::endl; // 7
    std::cout << s4 << " = " << calc.calculate(s4) << std::endl; // 13

    return 0;
}
```

### 这段代码怎么讲

#### 1）为什么这样分层

因为四则运算有优先级：

- `+ -` 优先级低
- `* /` 优先级高
- 数字 / 括号 / 一元符号放在最底层

#### 2）为什么支持括号

括号里的内容本身又是一个完整表达式，所以递归调用 `parseExpression()`。

#### 3）为什么能处理 `-3 + 5`

因为在 `factor` 层专门处理了一元 `+/-`。

#### 4）时间复杂度

每个字符基本只处理一次：

- 时间复杂度：`O(n)`
- 空间复杂度：递归栈最坏 `O(n)`

### 面试补充一句

> 如果题目不要求括号，也可以用“一个栈 + 当前数字 + 上一个运算符”的写法，代码更短；如果要求支持括号和一元负号，递归下降解析更清晰，扩展性更好。

## 最后背诵提纲

### TCP / UDP / IP

- IP：网络层，负责寻址和路由，不可靠
- TCP：传输层，面向连接，可靠，有序，字节流
- UDP：传输层，无连接，不可靠，报文，实时性好

### IO 多路复用

- 一个线程监听多个 fd
- select/poll 需要遍历全部 fd
- epoll 只返回就绪 fd，更适合高并发

### 程序到可执行文件

- 预处理
- 编译
- 汇编
- 链接

### 动态库和静态库

- 静态库：链接时拷贝进去
- 动态库：运行时加载

### 内存对齐

- 为了效率和硬件要求
- 结构体大小受成员顺序和最大对齐影响

### 虚函数大小

- 通常多一个 vptr
- 64 位一般 8 字节
- 最终大小还要看对齐

### 智能指针

- `unique_ptr`：独占
- `shared_ptr`：共享
- `weak_ptr`：弱引用，解决循环引用

### `enable_shared_from_this`

- 在对象内部安全获取自己的 `shared_ptr`
- 避免 `shared_ptr(this)` 重复释放

### `unique_ptr` 返回值

- 可以
- 依赖移动语义 / 返回值优化

### 互斥锁 + 条件变量

- `mutex` 保护共享资源
- `condition_variable` 用于同步，避免忙等

### `future` / `packaged_task`

- `future` 获取异步结果
- `packaged_task` 包装任务并把结果关联到 `future`

### `memory_order`

- 控制原子操作的顺序和可见性
- 常见：`relaxed` / `acquire` / `release` / `seq_cst`
