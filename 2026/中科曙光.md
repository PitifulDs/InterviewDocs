## 2. C++ 三大特性及理解

C++ 的核心在于其面向对象（OOP）的思想：

- **封装 (Encapsulation)：** 将数据（属性）和操作数据的函数（方法）绑定在一起，隐藏内部实现细节。
    - *理解：* 就像驱动汽车，你只需踩油门，不需要知道发动机活塞是如何运动的。
- **继承 (Inheritance)：** 允许一个类继承另一个类的属性和方法。
    - *理解：* 提高代码复用性，实现“is-a”的关系（如：狗是动物）。
- **多态 (Polymorphism)：** 允许不同对象对同一消息做出不同响应。主要通过虚函数（Virtual Function）实现。
    - *理解：* 同一个“叫”的指令，猫会“喵”，狗会“汪”。

## 5. 动态分配内存函数

- **C 语言：** `malloc()` 和 `free()`。
- **C++ 语言：** `new` 和 `delete`（对于数组使用 `new[]` 和 `delete[]`）。
    - *区别：* `new` 会调用构造函数，`delete` 会调用析构函数，而 `malloc` 仅仅是分配内存块。

## 6. static 关键字的作用

`static` 的核心在于控制 **生命周期** 和 **可见性**：

1. **局部变量：** 延长生命周期至程序结束，但作用域仍限制在函数内。
2. **全局变量/函数：** 限制链接性，使其仅在当前文件可见，防止命名冲突。
3. **类成员变量：** 所有对象共享同一份数据。
4. **类成员函数：** 无需实例化对象即可通过类名调用，不能直接访问非静态成员。

## 7. C 语言代码实现

### 冒泡排序

```cpp
void bubbleSort(int arr[], int n) {
    for (int i = 0; i < n - 1; i++) {
        for (int j = 0; j < n - 1 - i; j++) {
            if (arr[j] > arr[j + 1]) {
                int temp = arr[j];
                arr[j] = arr[j + 1];
                arr[j + 1] = temp;
            }
        }
    }
}
```

### n 的阶乘 (递归实现)

```cpp
long long factorial(int n) {
    if (n <= 1) return 1;
    return n * factorial(n - 1);
}
```

## 9. 内存泄漏怎么造成的

简单来说，就是**申请了内存却没释放**。

- **场景：** 使用 `new`/`malloc` 后忘记调用 `delete`/`free`。
- **异常导致：** 程序中途报错退出，跳过了释放逻辑。
- **对策：** 现代 C++ 建议使用**智能指针**（`std::unique_ptr`, `std::shared_ptr`）来自动管理内存。

## 10. Linux 进程间通信 (IPC)

主要有以下 6 种方式：

1. **管道 (Pipe)：** 匿名管道（父子进程）和命名管道（FIFO）。
2. **消息队列 (Message Queue)：** 消息链表。
3. **共享内存 (Shared Memory)：** 最快的方式，但需要同步机制。
4. **信号量 (Semaphore)：** 用于解决同步与互斥（常配合共享内存使用）。
5. **信号 (Signal)：** 唯一的异步通信。
6. **套接字 (Socket)：** 可用于跨机器通信。

## 11. 贪吃蛇应该用什么数据结构？

建议使用 **双向链表 (std::list)** 或 **双端队列 (std::deque)**。

- **原因：** 贪吃蛇运动的本质是“头加一个节，尾掉一个节”。
- **效率：** 这两种结构在头部和尾部插入/删除操作的时间复杂度都是 O(1)，非常高效。

## 7. 堆 (Heap) 和 栈 (Stack) 的区别

这是面试的高频考点，理解它们的管理方式是写好代码的基础。

| 特性 | 栈 (Stack) | 堆 (Heap) |
| --- | --- | --- |
| **分配方式** | 由编译器自动分配和释放。 | 由程序员手动申请和释放（`new`/`malloc`）。 |
| **空间大小** | 较小（通常几 MB），连续内存。 | 很大，取决于物理内存，不连续。 |
| **增长方向** | 高地址向低地址增长（向下）。 | 低地址向高地址增长（向上）。 |
| **速度** | 极快。 | 较慢，容易产生内存碎片。 |

**什么时候用？**

- **用栈：** 局部变量、函数参数。只要数据量不大且生命周期随函数结束，优先用栈。
- **用堆：** 需要动态确定大小（如用户输入的数组长度）、需要在函数结束后继续存在的数据。

## 8. 中断如何配置 (以嵌入式开发为例)

中断配置通常遵循以下“四步走”流程：

1. **初始化硬件：** 配置触发源（如 GPIO 引脚、定时器）。
2. **设置触发方式：** 比如上升沿、下降沿或低电平触发。

4. **编写 ISR（中断服务程序）：** 在特定的函数中编写逻辑，执行完后务必清掉中断标志位。

## 9. `i++` 和 `++i`

- **`++i` (前置)：** 先加 1，再返回。直接对变量操作，效率通常更高。
- **`i++` (后置)：** 先返回旧值，再加 1。系统需要产生一个临时变量来保存旧值，开销稍大。
- **口诀：** “前置加加性能好，后置加加有副本”。

## 10. 内存泄漏：产生与避免

- **产生原因：**
    - `malloc` 后没有 `free`。
    - 在循环或长生命周期的函数中不断申请内存却不释放。
    - 父类析构函数没有设为 `virtual`，导致子类部分内存没被释放。
- **如何避免：**
    - **养成习惯：** 申请和释放配对编写。
    - **RAII 机制：** 利用对象的生命周期管理资源。
    - **智能指针：** 使用 `std::unique_ptr` 或 `std::shared_ptr`（最推荐）。
    - **工具检测：** 使用 Valgrind 或 AddressSanitizer 进行静态/动态扫描。

## 11. 野指针：产生与避免

**野指针**是指向“不可用”内存区域的指针。

- **产生原因：**
    1. **定义未初始化：** 指针刚创建时不会自动指向 `NULL`，它会随机指向一个地址。
    2. **释放后未置空：** `free` 或 `delete` 之后，指针变量本身还保存着原来的地址，但内存已不可用。
    3. **指针越界：** 操作超出了数组或分配空间的范围。
- **如何避免：**
    1. **初始化：** 定义指针时一定要初始化为 `NULL` 或具体地址。
    2. **释放即置空：** 释放内存后立即执行 `ptr = NULL;`。
    3. **智能指针：** 再次强烈推荐，它们能极大地减少手动管理指针带来的风险。

## 1. 智能指针：`shared_ptr` 能否用在构造函数中？

这是个非常经典且阴险的陷阱题。

- **结论：** 可以直接传进去，但 **绝对不能** 在构造函数内部用 `this` 指针去构造一个 `shared_ptr`（或者调用 `shared_from_this()`）。
- **原因：** `shared_ptr` 的核心是引用计数。在构造函数执行期间，对象还没有完全“出生”（构造未完成），此时如果你尝试基于 `this` 创建 `shared_ptr`，引用计数无法正确管理这个尚未完全存在的对象，极易导致**重复释放**或**崩溃**。
- **标准做法：** 如果需要在类内部获取自身的 `shared_ptr`，必须让类继承 `std::enable_shared_from_this<T>`，且只能在构造函数**之后**（如初始化完成后的某个成员函数里）调用 `shared_from_this()`。

## 2. Lambda 表达式：值传递是如何进行的？

Lambda 的值捕获 `[=]` 实际上是**编译器自动生成了一个匿名类（闭包）**。

- **过程：** 当你选择值传递时，编译器会将捕获的变量作为该匿名类的**成员变量**。
- **拷贝时机：** 在 Lambda **定义**的那一刻，变量的值就被拷贝进了这个匿名类对象中。
- **注意：** 默认情况下，这些捕获的成员变量是 `const` 的。如果你想在 Lambda 内部修改它们，需要加上 `mutable` 关键字（但这不会影响外部原始变量的值）。

## 3. 左值引用 (Lvalue Reference)

- **定义：** 对一个“有名字、有固定内存地址”的对象的别名。
- **语法：** `T &ref = obj;`
- **本质：** 它就像是一个变量的“绰号”。左值引用必须在定义时初始化，且一旦绑定就不能更改指向。
- **对比：** 相对的是右值引用 `T &&`，后者专门用来接纳即将销毁的“临时工”（如函数返回值、字面量）。

## 1. `map` 的底层实现

C++ 标准库中的 `std::map` 和 `std::set` 底层通常是由 **红黑树 (Red-Black Tree)** 实现的。

- **特性：** 红黑树是一种自平衡的二叉查找树。它通过在节点上增加颜色属性（红或黑）以及一系列旋转和变色规则，确保没有一条路径会比其他路径长出两倍。
- **复杂度：** 无论插入、删除还是查找，时间复杂度都稳定在 O(logn)。
- **有序性：** 因为是二叉查找树，所以 `map` 里的元素是根据 Key 自动排序的。
*(注：如果是 `std::unordered_map`，底层则是哈希表，查找复杂度通常为 O(1)。)*

## 2. `vector` 和 `list` 的底层

这是两种截然不同的内存管理哲学：

### `vector`（动态数组）

- **底层：** 一块**连续**的内存空间。它内部通常维护三个指针：`start`（空间头）、`finish`（有效元素尾）、`end_of_storage`（空间尾）。
- **扩容机制：** 当空间满了再去添加元素时，`vector` 会申请一块更大的新内存（通常是原大小的 1.5 倍或 2 倍），把老数据全部拷贝过去，然后销毁老内存。这个操作代价很高。
- **优势与劣势：** 内存连续，支持 O(1) 随机访问，对 CPU 缓存极度友好；但在中间或头部插入/删除元素极其低效 O(n)。

### `list`（双向链表）

- **底层：** 一系列**不连续**的节点，每个节点包含数据以及指向前一个和后一个节点的指针（`prev` 和 `next`）。
- **扩容机制：** 每次添加新元素就单独 `new` 一个节点并用指针连接，不存在整体拷贝和扩容的问题。
- **优势与劣势：** 只要拿到了迭代器，在任何位置插入和删除都是 O(1) 的；但不支持随机访问，找第 n 个元素必须从头遍历 O(n)，且内存不连续，容易产生内存碎片。

## 3. `allocator`（空间配置器）的底层实现

STL 默认的 `allocator` 核心是为了解决**频繁分配小内存带来的性能消耗和内存碎片问题**。以经典的 SGI STL 为例，它采用了**双层级配置器**：

- **第一级配置器：** 直接调用 C 语言的 `malloc()` 和 `free()`。适用于申请大于 128 Bytes 的大内存。它还包含一套内存耗尽（Out of Memory）时的处理机制。
- **第二级配置器（核心）：** 采用**内存池 (Memory Pool)** 和**自由链表 (Free List)**。
    - 它维护了 16 个自由链表，分别管理大小为 8, 16, 24, ..., 128 Bytes 的小内存块。
    - 当你申请小内存时，它会自动把大小向上对齐到 8 的倍数，然后直接从对应的自由链表里“拔”一块给你。释放时再“插”回去。
    - **意义：** 极大地减少了调用操作系统分配内存的开销，并有效避免了外碎片。

## 4. 编译链接过程及“找不到符号”问题

C++ 的生成过程分为四步：**预处理 (Preprocessing) → 编译 (Compilation) → 汇编 (Assembly) → 链接 (Linking)**。

面试官问的**“找不到符号”（Undefined reference to / Unresolved external symbol）**，明确发生在这四步中的最后一步：**链接阶段**。

**常见原因：**

1. **只声明未定义：** 在头文件中写了函数声明，但没有在 `.cpp` 文件中写具体实现。
2. **忘记链接库：** 你的代码用到了第三方库（如 pthread、opencv），但在编译命令或 CMake 中没有指定链接该库（如 GCC 漏了 `-l` 参数）。
3. **C/C++ 混编问题：** C++ 支持函数重载，编译器会对函数名进行“名字修饰”（Name Mangling）。如果你在 C++ 中调用 C 语言编译的库，名字匹配不上就会报错。此时需要用 `extern "C"` 包裹 C 语言的头文件。
4. **链接顺序错误：** 依赖库放在了被依赖库的前面，导致链接器往后找符号时遗漏。

## 5. 函数调用过程：栈与寄存器的调整

这是操作系统和汇编层面的硬核知识。每次函数调用，系统都会在内存的栈区为这个函数分配一块专属空间，叫做 **栈帧 (Stack Frame)**。

假设函数 A 调用函数 B，核心过程如下（以 x86 架构的 `ebp`/`esp` 寄存器为例）：

1. **传参（A 做）：** A 将传给 B 的参数压入栈中（通常是从右向左压栈）。
2. **保存返回地址（A 做）：** 执行 `call` 指令，把 A 中下一条要执行的指令地址压入栈，以便 B 执行完能找回来。
3. **准备新栈帧（B 做）：**
    - `push ebp`：保存 A 的栈底指针。
    - `mov ebp, esp`：将 B 的栈底指针指向当前栈顶，标志着 B 的栈帧正式建立。
    - `sub esp, xxx`：把栈顶指针往上移动（减去一个值），为 B 的局部变量腾出空间。
4. **执行逻辑：** B 执行自己的代码。
5. **销毁栈帧并返回（B 做）：**
    - `mov esp, ebp`：直接把栈顶拉回 B 的栈底，瞬间销毁所有局部变量空间。
    - `pop ebp`：把之前保存的 A 的栈底指针恢复。
    - `ret`：弹出之前保存的返回地址，跳回函数 A 继续执行。

### 1. 指针和引用的区别

- **本质区别：** 指针是一个独立的实体（变量），它在内存中占有空间，里面存的是别人的内存地址；引用只是一个目标变量的“别名”，在语言层面它和目标变量就是同一个东西。
- **初始化：** 指针可以不初始化（但不推荐，会变野指针），也可以随时重新赋值指向其他地址；引用**必须在定义时初始化**，且一旦绑定就**永远不能再改变**绑定的对象。
- **空值：** 指针可以为 `nullptr` 或 `NULL`；引用不能绑定到空值。
- **多级结构：** 指针有多级指针（如 `int** p`）；引用只有一级，不存在“引用的引用”。

### 2. 程序崩溃怎么调试？

面试官想听的是一套完整的排查方法论：

1. **日志排查 (Log)：** 查看程序崩溃前最后打印的日志，缩小案发范围。
2. **GDB 调试：** 确保编译时加了 `-g` 选项。如果是必现的崩溃，可以直接 `gdb ./执行文件` 跑起来复现；如果是偶发卡死，可以用 `gdb attach <PID>` 挂载到运行的进程上分析。
3. **分析 Core Dump：** 最常用的事后分析法（详见第 7 问）。
4. **内存检测工具：** 很多崩溃是内存乱踩导致的，可以借助工具提前暴露。比如使用 Valgrind，或者在编译时加入 AddressSanitizer (`-fsanitize=address`)。

### 3. std::shared_ptr 的理解

- **核心机制：** 采用**引用计数**。多个 `shared_ptr` 可以共享同一块内存，每次拷贝，引用计数 +1；每次销毁（跳出作用域），引用计数 -1。当计数归 0 时，自动调用 `delete` 释放内存。
- **线程安全性：** `shared_ptr` 的引用计数增减是原子的（线程安全），但**它管理的对象本身不是线程安全的**。多线程读写同一个被管理的对象仍需手动加锁。
- **致命弱点：循环引用。** 如果对象 A 中有指向对象 B 的 `shared_ptr`，同时 B 中也有指向 A 的 `shared_ptr`，它们的引用计数永远降不到 0，导致内存泄漏。**解决办法**是使用 `std::weak_ptr` 来打破循环。

### 4. CMake 怎么指定动态编译？

- **生成动态库：** 在 CMakeLists.txt 中使用 `add_library`，并明确指定 `SHARED` 关键字。
    - `add_library(MyLib SHARED src.cpp)`
- **链接动态库：** 使用 `target_link_libraries` 将其链接到执行文件或其他库。
    - `add_executable(MyApp main.cpp)`
    - `target_link_libraries(MyApp PRIVATE MyLib)`
- *(补充技巧)* 如果想全局默认生成动态库，可以在顶层加入：`set(BUILD_SHARED_LIBS ON)`。

### 5. 编译成功，但报错 Undefined reference（找不到符号），怎么处理？

“编译成功”意味着单文件 `.cpp` 转 `.o` 没问题，报错发生在**链接阶段**。排查思路如下：

1. **检查库是否链接：** CMake 中是否漏写了 `target_link_libraries`，导致压根没去链接对应的库。
2. **链接顺序错乱：** 链接器（如 gcc）默认是从左向右解析符号的。依赖别人的库要放在前面，被依赖的底层库要放在后面。
3. **C/C++ 混编问题：** C++ 支持函数重载，会对函数名进行“名字修饰”（Name Mangling）。如果链接纯 C 语言编译的库，头文件中必须加 `extern "C"` 才能匹配上符号。
4. **只有声明没有实现：** 检查报错的那个函数，是不是在头文件里声明了，但对应的 `.cpp` 源文件忘记加进 CMake 编译列表里了。

### 6. 定位到某个 .so 动态库找不到符号链接，怎么处理？

这通常是运行期或动态链接时的问题。

1. **查缺失的依赖：** 使用 `ldd libxxx.so` 命令。如果输出里有 `not found`，说明它依赖的其他动态库找不到了。通常是因为环境变量 `LD_LIBRARY_PATH` 没配置好，或者库缺失。
2. **查具体的符号：** 使用 `nm -C libxxx.so | grep 缺失的符号名`（`-C` 用来还原 C++ 的函数名）。如果输出的前缀是 `U`，代表 Undefined（未定义），说明这个 `.so` 确实缺少这个实现，需要检查编译这个 `.so` 时的依赖关系。
3. **ABI 兼容问题：** 如果符号名差了一点点（比如包含 cxx11 字眼），通常是不同编译器版本（如老旧的 GCC 4.8 和新版 GCC）导致的 C++ ABI 不兼容，需要统一部署环境的编译器版本。

### 7. 生成的 Core 文件可以输出什么信息？

Core 文件是程序崩溃瞬间的“内存快照”及寄存器状态。配合 `gdb ./程序 ./core文件` 使用。

- **调用栈 (Call Stack)：** 输入 `bt` (backtrace)，这是最关键的，直接打印出崩溃时是由哪些函数一层层调用的，准确到具体哪一行代码崩的。
- **局部变量状态：** 用 `frame N` 切换到某一层函数，然后输入 `info locals` 查看当时的局部变量，往往能发现奇怪的越界值或空指针。
- **特定变量值：** 输入 `p 变量名` 可以打印当时指针的指向或对象的内部数据。
- **寄存器状态：** 输入 `info registers` 查看 CPU 寄存器（如段错误往往伴随无效的指令指针 EIP/RIP）。
- **内存原始数据：** 使用 `x/参数 内存地址` 可以打印特定地址段的十六进制内存数据，用于排查底层内存踩踏。

### 1、常见的数据结构（类型）所占字节

这里通常指基本数据类型。面试时一定要先声明前提条件：**“在 64 位 Linux（LP64）系统下”**，因为不同架构会有所不同。

- `char`: 1 字节
- `short`: 2 字节
- `int`: 4 字节
- `float`: 4 字节
- `double`: 8 字节
- `long`: 8 字节（注意：在 32 位系统或 64 位 Windows 中通常是 4 字节）
- `long long`: 8 字节
- **指针 (如 `int*`, `void*`)**: 8 字节（32位系统下是 4 字节）

### 2、结构体数据存储占内存问题（内存对齐）

这是高频考点。编译器为了提高 CPU 的内存读取效率，会进行“内存对齐”。

**对齐规则：结构体的总大小必须是最宽基本类型成员大小的整数倍；每个成员的偏移量也必须是自身大小的整数倍（不足则补齐 padding）。**

- **第一种：`struct { int a; int b; double c; }`**
    - `int a` 占 4 字节（偏移量 0-3）。
    - `int b` 占 4 字节（偏移量 4-7，刚好对齐）。
    - 此时共 8 字节。
    - `double c` 占 8 字节（偏移量 8-15，8 是 8 的倍数，不需要补齐）。
    - **总大小：16 字节。**
- **第二种：`struct { int a; double b; int c; }`**
    - `int a` 占 4 字节（偏移量 0-3）。
    - `double b` 占 8 字节。为了让它的偏移量是 8 的倍数，`a` 后面必须补齐 4 个字节的空白。于是 `b` 占据偏移量 8-15。
    - 此时共 16 字节。
    - `int c` 占 4 字节（偏移量 16-19）。
    - 检查总大小：目前是 20 字节。由于结构体最宽类型是 `double`（8 字节），总大小必须是 8 的倍数，所以要在末尾再补 4 字节。
    - **总大小：24 字节。**

### 3、什么时候用 `union` (联合体)？

`union` 的所有成员**共享同一块内存**，其大小等于最大成员的大小。改变一个成员，其他成员的值也会被破坏。

**使用场景：**

1. **节省内存：** 当几个数据项互斥，绝对不会同时出现时（例如网络协议报文的不同类型载荷，或者设备状态机）。
2. **底层数据转换（类型惩罚 Type Punning）：** 比如把一个 `float` 按位解析成 `int`，或者分离高低字节。

### 4、什么是指针？指针的优势

- **定义：** 指针是一个存放内存地址的变量。
- **优势：**
    1. 直接操纵硬件/内存：对底层开发至关重要。
    2. 提升效率：函数传参时传递指针（通常 8 字节）远比拷贝整个庞大的结构体或类要快得多。
    3. 动态内存分配：配合 `new`/`malloc` 在堆区管理生命周期不受作用域限制的内存。
    4. 构建复杂数据结构：链表、树、图都依赖指针连接。

### 5、进程与线程的区别

- **资源分配：** 进程是操作系统资源分配的基本单位。线程是 CPU 调度的基本单位。
- **内存共享：** 同一进程内的线程共享代码段、数据段和堆，但每个线程有独立的栈和寄存器上下文。
- **开销：** 进程切换开销大；线程更轻量。
- **健壮性：** 线程崩溃可能导致整个进程一起崩溃。

### 6、线程与 CPU 的关系

CPU 调度的最小执行流其实就是线程。多核可真正并行，单核通过时间片并发。

### 7、两个线程同时进行 100 次 `a++`，结果是什么？（从内存角度出发）

- **结果：** 最终结果通常 **<= 200**（会丢失更新）。
- **原因：** `a++` 不是原子操作（load/add/store），可能发生竞态。

### 8、线程通信方式

共享内存天然可通信，但必须配合同步机制：互斥锁、条件变量、信号量、读写锁。

### 9、自旋锁 (Spinlock)

忙等获取锁，适合临界区很短；否则浪费 CPU。

- Linux：`top`、`ps -ef`、`netstat -ntlp`、`tail -f`
- GDB：`b`、`r`、`c`、`n`、`s`、`p`、`bt`

- PCB（`task_struct`）在内核空间。

### 3. `static` 与 `const` 关键字

- **`static` (控制可见性与生命周期)：**
    - **局部变量：** 存在静态存储区，函数调用结束后不销毁，下次调用保持原值。
    - **全局变量/函数：** 限制为“内部链接”，只能在当前文件内访问，防止命名冲突。
    - **类成员：** 静态成员变量所有对象共享；静态成员函数没有 `this` 指针，只能访问静态变量。
- **`const` (控制只读特性)：**
    - **变量：** 声明为常量，必须在定义时初始化。
    - **指针：** 区分常量指针 (`const int* p`，指向的值不可变) 和指针常量 (`int* const p`，指针的指向不可变)。
    - **类成员函数：** 放在函数末尾（如 `void func() const;`），承诺不修改任何成员变量。

### 4. 单例设计模式 (Singleton)

确保一个类只有一个实例，并提供一个全局访问点。

- **现代 C++ 最佳实践（Meyers' Singleton）：** 利用 C++11 标准保证的“局部静态变量初始化是线程安全的”特性。

```cpp
class Singleton {
public:
    static Singleton& getInstance() {
        static Singleton instance; // 线程安全的局部静态变量
        return instance;
    }
private:
    Singleton() {} // 构造函数私有化
    Singleton(const Singleton&) = delete; // 禁用拷贝
    Singleton& operator=(const Singleton&) = delete; // 禁用赋值
};
```

### 5. 静态链接库 (`.a`/`.lib`) 与动态链接库 (`.so`/`.dll`)

| 特性 | 静态链接库 | 动态链接库 |
| --- | --- | --- |
| **链接时机** | 编译时。代码会被完整拷贝进可执行文件中。 | 运行时。可执行文件只记录符号，运行时加载库。 |
| **文件大小** | 生成的可执行文件很大。 | 生成的可执行文件小。 |
| **内存开销** | 多个程序使用同一个静态库时，内存中有多个副本。 | 多个程序可以共享物理内存中的同一个动态库副本。 |
| **升级维护** | 库更新后，所有依赖程序必须**重新编译**。 | 只需替换库文件即可，依赖程序无需重新编译。 |

### 6. SPI 总线协议

SPI 是一种全双工、同步、主从架构的串行通信协议，常用于嵌入式开发。

- **四根线：** SCLK（时钟）、MOSI（主出从入）、MISO（主入从出）、CS/SS（片选信号）。
- **特点：** 速度快于 I2C，但没有指定的寻址机制，完全靠拉低硬件片选线（CS）来选择从机。

### 7 & 8. HTTP vs HTTPS 及 HTTPS 的工作量

- **区别：** HTTP 是明文传输，端口 80；HTTPS = HTTP + SSL/TLS 加密层，端口 443，能够防止中间人窃听和篡改。
- **服务器支持 HTTPS 需要增加的工作量：**
    1. **证书申请：** 向 CA 机构申请 SSL 证书（包含公钥）并部署在服务器上。
    2. **服务器配置：** 修改 Nginx/Apache 配置，开启 443 端口，加载证书和私钥，通常还要配置 80 端口强制重定向到 443。
    3. **性能开销考量：** TLS 握手涉及非对称加密，非常消耗 CPU。可能需要考虑硬件加速或优化 Session 缓存。

### 9. IPv4 与 IPv6 的区别

1. **地址空间：** IPv4 是 32 位（约 43 亿地址）；IPv6 是 128 位，号称“能给地球上每一粒沙子分配一个 IP”。
2. **报文头部：** IPv6 头部更精简，去掉了校验和，路由器转发速度更快。
3. **NAT 需求：** IPv4 极度依赖 NAT (网络地址转换) 续命；IPv6 地址管够，设备可以直接暴露在公网，无需 NAT。

### 10. 两台内网主机如何感知对方地址并通信？ (NAT 穿透)

这是一个典型的 **P2P 通信 / NAT 打洞 (Hole Punching)** 问题。

- **机制：** 双方都需要先连接到一个**公网的 STUN 服务器**。STUN 服务器能够识别出双方各自经过 NAT 转换后的“公网 IP 和端口”。然后，服务器将这些信息交换给双方。双方拿着对方的公网 IP 和端口直接发送 UDP 数据包，从而“打穿” NAT 建立直连通道。如果 NAT 类型极其严格（对称型），则只能回退使用 TURN 服务器进行数据中转。

### 11. TCP 三次握手与四次挥手

- **握手（建立连接）：** `SYN` → `SYN-ACK` → `ACK`。核心目的是同步双方的初始序列号 (Sequence Number)，并确认双方的收发能力都正常。
- **挥手（断开连接）：** `FIN` → `ACK` → `FIN` → `ACK`。为什么是四次？因为 TCP 是全双工的，允许“半关闭”（我发完数据了，不发了，但我还能接收你的数据）。

### 12 & 13. 没有 IP 能通信吗？如何基于 IP 获取 MAC？

- **能通信。** 如果两台主机在**同一个局域网（同一台交换机下）**，完全可以不依赖 IP 层（网络层），直接基于 **MAC 地址（数据链路层）**进行以太网帧的广播和通信。
- **协议：** 使用 **ARP (地址解析协议)** 可以通过广播询问局域网内的设备：“谁拥有这个 IP 地址？请把你的 MAC 地址告诉我。”

### 14. 多个连接同时请求同一个文件资源，怎么处理？

这是一道优秀的系统设计题，考察你对高并发 I/O 的理解：

1. **内存缓存 (Cache)：** 绝对不能让每个请求都去读磁盘。应该将文件读入内存（如 Redis，或进程内本地 LRU Cache），后续请求直接打到内存。
2. **零拷贝技术 (Zero-Copy)：** 使用 Linux 的 `sendfile` 系统调用。数据直接从内核的文件缓冲区拷贝到网卡 Socket 缓冲区，跳过了拷贝到用户态的步骤，极大降低 CPU 占用。
3. **并发控制 (读写锁)：** 如果这个文件资源会被动态修改，为了防止读到脏数据，需要使用读写锁（如 C++17 的 `std::shared_mutex`），允许多个线程并发读，但写时独占。

### 15. Linux 内存管理

面试官想听的宏观机制：

- **虚拟内存：** 每个进程都有独立的虚拟地址空间（32位下 4GB），通过 **页表 (Page Table)** 和 MMU 硬件映射到物理内存。
- **物理内存管理：** * **伙伴系统 (Buddy System)：** 负责分配连续的物理内存页，防止外部碎片。
    - **Slab 分配器：** 负责分配内核中的小块对象内存，防止内部碎片。

### 16. 线程池 (Thread Pool)

- **目的：** 避免频繁创建和销毁线程带来的内核态切换开销。
- **核心结构：** 1. **任务队列 (Task Queue)：** 存放待处理任务，通常需要互斥锁保护。
2. **工作线程组 (Worker Threads)：** 一批预先创建好的线程，死循环从队列中取任务执行。
3. **同步机制：** 当队列为空时，线程使用**条件变量 (Condition Variable)** 挂起休眠；当有新任务到来时，唤醒线程工作。

### 17. 64 位系统与 32 位系统的区别

- **寻址能力：** 32 位系统最多支持 232 字节（即 4GB）物理内存；64 位系统理论上支持 264 字节（极其庞大）。
- **指针大小：** 32 位系统指针占 4 字节；64 位系统指针占 8 字节。
- **运算效率：** 64 位 CPU 的通用寄存器更宽，一次可以处理 64 位的数据（如长整型或双精度浮点数），计算密集型任务效率更高。
