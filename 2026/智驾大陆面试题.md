## 第一部分：线性结构、STL 容器、线程安全

### 1. 队列的概念

#### 1.1 什么是队列

队列（Queue）是一种**先进先出（FIFO, First In First Out）** 的线性数据结构。

- 插入元素的一端叫 **队尾**（rear / tail）
- 删除元素的一端叫 **队头**（front / head）

也就是说：
- 先进去的元素先出来
- 后进去的元素后出来

这和现实中的排队非常像。

#### 1.2 队列的典型操作

常见操作包括：

- `push`：入队
- `pop`：出队
- `front`：查看队头元素
- `empty`：判断是否为空
- `size`：当前元素个数

#### 1.3 队列的应用场景

队列在工程中非常常见，比如：

- 线程池中的任务队列
- 消息队列
- BFS 广度优先搜索
- 网络请求缓冲
- 日志异步落盘
- 生产者-消费者模型

#### 1.4 面试可直接说的版本

> 队列是一种先进先出的线性结构，元素从队尾插入，从队头删除。它非常适合处理按到达顺序消费的数据，比如线程池任务调度、消息缓冲、生产者消费者模型等。常见实现方式有顺序存储和链式存储。

### 3. 用数组实现队列

#### 3.1 普通顺序队列的问题

如果直接用数组，队头不断后移，会导致前面的空间无法复用。

例如：

```text
[ _, _, 3, 4, 5 ]
```

虽然前面有空位，但尾指针可能已经到末尾了。

所以工程里一般使用：

#### 3.2 环形队列（推荐）

核心思想：

- 把数组首尾连接起来看
- `tail` 到底后可以回到 0
- 用取模实现循环利用空间

#### 3.3 代码示例：环形队列

```cpp
#include <iostream>
#include <stdexcept>

class CircularQueue {
public:
    explicit CircularQueue(size_t capacity)
        : data_(new int[capacity + 1]), capacity_(capacity + 1), head_(0), tail_(0) {}

    ~CircularQueue() {
        delete[] data_;
    }

    bool push(int value) {
        if (isFull()) {
            return false;
        }
        data_[tail_] = value;
        tail_ = (tail_ + 1) % capacity_;
        return true;
    }

    bool pop() {
        if (empty()) {
            return false;
        }
        head_ = (head_ + 1) % capacity_;
        return true;
    }

    int front() const {
        if (empty()) {
            throw std::runtime_error("queue is empty");
        }
        return data_[head_];
    }

    bool empty() const {
        return head_ == tail_;
    }

    bool isFull() const {
        return (tail_ + 1) % capacity_ == head_;
    }

    size_t size() const {
        return (tail_ + capacity_ - head_) % capacity_;
    }

private:
    int* data_;
    size_t capacity_;
    size_t head_;
    size_t tail_;
};
```

#### 3.4 面试要点

> 如果用数组实现队列，普通顺序队列会有“假溢出”问题，所以通常会用环形队列，通过 head/tail 配合取模操作复用空间。

### 4. 用链表实现队列

#### 4.1 思路

链表实现队列时：

- 用 `head` 指向队头
- 用 `tail` 指向队尾
- 入队：尾插
- 出队：头删

#### 4.2 代码示例

```cpp
#include <iostream>
#include <stdexcept>

class LinkedQueue {
private:
    struct Node {
        int val;
        Node* next;
        explicit Node(int v) : val(v), next(nullptr) {}
    };

public:
    LinkedQueue() : head_(nullptr), tail_(nullptr), size_(0) {}

    ~LinkedQueue() {
        while (!empty()) {
            pop();
        }
    }

    void push(int value) {
        Node* node = new Node(value);
        if (tail_) {
            tail_->next = node;
        } else {
            head_ = node;
        }
        tail_ = node;
        ++size_;
    }

    void pop() {
        if (empty()) {
            throw std::runtime_error("queue is empty");
        }
        Node* temp = head_;
        head_ = head_->next;
        if (!head_) {
            tail_ = nullptr;
        }
        delete temp;
        --size_;
    }

    int front() const {
        if (empty()) {
            throw std::runtime_error("queue is empty");
        }
        return head_->val;
    }

    bool empty() const {
        return size_ == 0;
    }

    size_t size() const {
        return size_;
    }

private:
    Node* head_;
    Node* tail_;
    size_t size_;
};
```

#### 4.3 数组队列 vs 链表队列

| 对比项 | 数组 / 环形队列 | 链表队列 |
|---|---|---|
| 内存连续 | 是 | 否 |
| 随机访问 | 可以 | 不可以 |
| 扩容 | 可能需要搬迁 | 不需要整体搬迁 |
| 单个节点额外开销 | 无 | 有指针开销 |
| 缓存友好 | 更好 | 较差 |

### 5. 针对队列延伸：vector 和链表的问题

#### 5.1 `vector` 的特点

- 底层是**连续内存**
- 支持随机访问，`O(1)`
- 尾插效率高，均摊 `O(1)`
- 中间插入/删除效率低，需要搬移元素
- 扩容时可能整体拷贝/移动

#### 5.2 链表的特点

- 内存不连续
- 不支持随机访问
- 已知位置插入删除快
- 查找慢，缓存命中率低
- 每个节点有额外指针开销

#### 5.3 面试回答模板

> `vector` 适合读多写少、需要随机访问的场景；链表适合频繁插入删除、且已经拿到节点位置的场景。但在现代 CPU 下，因为缓存友好性，很多时候 `vector` 的实际表现比链表更好。

### 6. 如何保证队列的线程安全

#### 6.1 基本思路

多个线程同时操作队列，会出现：

- 数据竞争
- 队列状态被破坏
- 一个线程 `pop`，另一个线程同时 `push`

常见解决方法：

1. 用 **互斥锁 `std::mutex`** 保护临界区
2. 用 **条件变量 `std::condition_variable`** 实现阻塞等待
3. 更高阶方案：无锁队列 / CAS，但实现复杂

#### 6.2 线程安全队列示例

```cpp
#include <queue>
#include <mutex>
#include <condition_variable>
#include <stdexcept>

template <typename T>
class ThreadSafeQueue {
public:
    void push(const T& value) {
        {
            std::lock_guard<std::mutex> lock(mutex_);
            queue_.push(value);
        }
        cv_.notify_one();
    }

    void push(T&& value) {
        {
            std::lock_guard<std::mutex> lock(mutex_);
            queue_.push(std::move(value));
        }
        cv_.notify_one();
    }

    T wait_and_pop() {
        std::unique_lock<std::mutex> lock(mutex_);
        cv_.wait(lock, [this] { return !queue_.empty(); });

        T value = std::move(queue_.front());
        queue_.pop();
        return value;
    }

    bool try_pop(T& value) {
        std::lock_guard<std::mutex> lock(mutex_);
        if (queue_.empty()) {
            return false;
        }
        value = std::move(queue_.front());
        queue_.pop();
        return true;
    }

    bool empty() const {
        std::lock_guard<std::mutex> lock(mutex_);
        return queue_.empty();
    }

private:
    mutable std::mutex mutex_;
    std::queue<T> queue_;
    std::condition_variable cv_;
};
```

#### 6.3 面试回答模板

> 保证线程安全最直接的方法是使用互斥锁保护队列的入队和出队操作。如果需要消费者阻塞等待任务，可以再结合条件变量。这样就可以构成一个典型的生产者-消费者队列。

### 7. 互斥锁的用法

#### 7.1 为什么需要互斥锁

当多个线程访问同一份共享资源时，如果没有同步机制，就会产生**竞态条件**。

#### 7.2 常见锁类型

- `std::mutex`：普通互斥锁
- `std::recursive_mutex`：递归锁，**同一个线程可以对同一把锁重复加锁多次，而不会把自己阻塞住** 的互斥锁
- `std::timed_mutex`：带超时的锁
- `std::shared_mutex`（C++17）：读写锁

#### 7.3 两种常见写法

**写法 1：`lock()` / `unlock()`**

```cpp
std::mutex mtx;
int counter = 0;

void add() {
    mtx.lock();
    ++counter;
    mtx.unlock();
}
```

问题：异常时可能忘记解锁。

**写法 2：RAII 风格，推荐**

```cpp
std::mutex mtx;
int counter = 0;

void add() {
    std::lock_guard<std::mutex> lock(mtx);
    ++counter;
}
```

#### 7.4 `unique_lock` 和 `lock_guard` 区别

- `lock_guard`：轻量、简单、作用域结束自动解锁
- `unique_lock`：更灵活，支持延迟加锁、手动解锁、配合条件变量

```cpp
std::mutex mtx;
std::condition_variable cv;
bool ready = false;

void worker() {
    std::unique_lock<std::mutex> lock(mtx);
    cv.wait(lock, [] { return ready; });
}
```

#### 7.5 面试回答模板

> 互斥锁用于保护临界区，避免多个线程同时修改共享资源。工程里通常优先用 RAII 风格的 `lock_guard` 或 `unique_lock`，这样可以避免异常路径下忘记解锁的问题。

### 9. 智能指针

#### 9.1 为什么需要智能指针

裸指针的问题：

- 容易忘记 `delete`
- 容易重复释放
- 所有权不清晰
- 异常情况下资源容易泄漏

智能指针本质上是：

> **用一个对象管理资源，在对象析构时自动释放资源。**

这就是 RAII 思想的典型应用。

#### 9.2 三种常见智能指针

##### `std::unique_ptr`

- 独占所有权
- 不允许拷贝
- 允许移动
- 开销小

```cpp
std::unique_ptr<int> p = std::make_unique<int>(10);
```

##### `std::shared_ptr`

- 共享所有权
- 内部有引用计数
- 最后一个拥有者析构时释放资源

```cpp
std::shared_ptr<int> p1 = std::make_shared<int>(10);
std::shared_ptr<int> p2 = p1;
```

##### `std::weak_ptr`

- 不增加引用计数
- 用于打破循环引用
- 必须通过 `lock()` 转成 `shared_ptr`

```cpp
std::weak_ptr<int> wp = p1;
if (auto sp = wp.lock()) {
    std::cout << *sp << std::endl;
}
```

#### 9.3 `unique_ptr` 和 `shared_ptr` 的异同

| 对比项 | `unique_ptr` | `shared_ptr` |
|---|---|---|
| 所有权 | 独占 | 共享 |
| 拷贝 | 不允许 | 允许 |
| 移动 | 允许 | 允许 |
| 开销 | 小 | 较大（引用计数） |
| 适用场景 | 明确唯一所有者 | 多处共享对象 |

#### 9.4 “指针 a 赋值给指针 b 有什么后果”

这个问题要分情况。

##### 裸指针

```cpp
int* a = new int(10);
int* b = a;
```

后果：
- `a` 和 `b` 指向同一块内存
- 如果两个都 `delete`，会重复释放
- 如果一个释放后另一个继续使用，会变成悬空指针

##### `unique_ptr`

```cpp
std::unique_ptr<int> a = std::make_unique<int>(10);
// std::unique_ptr<int> b = a; // 编译错误
std::unique_ptr<int> b = std::move(a);
```

后果：
- 不能拷贝，只能移动
- 移动后 `a` 失去所有权，变为空

##### `shared_ptr`

```cpp
auto a = std::make_shared<int>(10);
auto b = a;
```

后果：
- 引用计数 +1
- 两者共同管理资源
- 最后一个析构时才释放

#### 9.5 面试回答模板

> 智能指针是 RAII 的典型应用，用对象管理堆资源，避免手工 `new/delete`。`unique_ptr` 表示独占所有权，适合单一拥有者场景；`shared_ptr` 表示共享所有权，底层通过引用计数管理生命周期；`weak_ptr` 用来观察 `shared_ptr` 管理的对象，常用于解决循环引用。

### 10. RAII 是什么

RAII 全称是 **Resource Acquisition Is Initialization**，即：

> 资源获取即初始化，资源释放由对象生命周期自动管理。

常管理的资源：

- 内存
- 文件句柄
- socket
- 锁
- 数据库连接

#### 10.1 示例：锁的 RAII

```cpp
std::mutex mtx;

void func() {
    std::lock_guard<std::mutex> lock(mtx);
    // 临界区
}
```

`lock_guard` 构造时加锁，析构时解锁。

#### 10.2 面试回答模板

> RAII 的核心思想是把资源绑定到对象生命周期上，在构造时获取资源，在析构时释放资源。这样能避免忘记释放和异常路径下泄漏。智能指针、锁封装、文件句柄封装，本质上都是 RAII。

### 11. 移动语义

#### 11.1 为什么需要移动语义

很多对象内部管理堆内存，比如 `string`、`vector`。如果每次返回对象都深拷贝，开销很大。

移动语义的核心就是：

> 不拷贝资源，而是把资源“转移”给新的对象。

#### 11.2 左值 / 右值

- **左值**：有名字、可取地址、可长期存在
- **右值**：临时对象、即将销毁的值

```cpp
int a = 10;  // a 是左值，10 是右值
```

#### 11.3 右值引用

```cpp
int&& x = 10;
```

右值引用用于绑定右值，从而触发移动语义。

#### 11.4 移动构造 / 移动赋值示例

```cpp
#include <iostream>
#include <cstring>

class MyString {
public:
    MyString() : data_(nullptr) {}

    explicit MyString(const char* s) {
        size_t len = std::strlen(s);
        data_ = new char[len + 1];
        std::memcpy(data_, s, len + 1);
    }

    // 拷贝构造
    MyString(const MyString& other) {
        if (other.data_) {
            size_t len = std::strlen(other.data_);
            data_ = new char[len + 1];
            std::memcpy(data_, other.data_, len + 1);
        } else {
            data_ = nullptr;
        }
    }

    // 移动构造
    MyString(MyString&& other) noexcept : data_(other.data_) {
        other.data_ = nullptr;
    }

    // 拷贝赋值
    MyString& operator=(const MyString& other) {
        if (this == &other) return *this;
        delete[] data_;
        if (other.data_) {
            size_t len = std::strlen(other.data_);
            data_ = new char[len + 1];
            std::memcpy(data_, other.data_, len + 1);
        } else {
            data_ = nullptr;
        }
        return *this;
    }

    // 移动赋值
    MyString& operator=(MyString&& other) noexcept {
        if (this == &other) return *this;
        delete[] data_;
        data_ = other.data_;
        other.data_ = nullptr;
        return *this;
    }

    ~MyString() {
        delete[] data_;
    }

private:
    char* data_;
};
```

#### 11.5 面试回答模板

> 移动语义是 C++11 引入的重要特性，它通过右值引用把资源从源对象转移到目标对象，而不是做深拷贝。这样可以大幅减少临时对象和返回值构造时的开销。常见形式就是移动构造和移动赋值。

### 12. `std::forward` 了解吗

#### 12.1 它解决什么问题

模板参数在转发过程中，可能丢失原来的值类别（左值 / 右值）。

`std::forward<T>(x)` 的作用是：

> 在模板中**保留参数原本的左值/右值属性**，实现完美转发。

#### 12.2 示例

```cpp
#include <iostream>
#include <utility>

void process(int& x) {
    std::cout << "lvalue\n";
}

void process(int&& x) {
    std::cout << "rvalue\n";
}

template <typename T>
void wrapper(T&& x) {
    process(std::forward<T>(x));
}

int main() {
    int a = 10;
    wrapper(a);   // lvalue
    wrapper(20);  // rvalue
}
```

#### 12.3 面试回答模板

> `std::forward` 主要用于模板中的完美转发。因为模板参数 `T&&` 在推导时可能同时接受左值和右值，`forward` 可以把参数按它原本的值类别继续传下去，避免把右值错误地当成左值。

### 13. lambda 表达式在函数中哪里使用

#### 13.1 本质

lambda 是一个**匿名函数对象**，适合写短逻辑、回调逻辑、局部策略。

#### 13.2 常见使用场景

1. STL 算法
2. 排序比较器
3. 回调函数
4. 线程入口
5. 局部临时逻辑

#### 13.3 示例

```cpp
#include <algorithm>
#include <vector>
#include <iostream>

int main() {
    std::vector<int> nums{4, 2, 5, 1, 3};

    std::sort(nums.begin(), nums.end(), [](int a, int b) {
        return a > b;
    });

    for (int n : nums) {
        std::cout << n << " ";
    }
}
```

#### 13.4 捕获列表

- `[ ]`：不捕获
- `[=]`：值捕获
- `[&]`：引用捕获
- `[this]`：捕获当前对象

#### 13.5 面试回答模板

> lambda 本质上是匿名函数对象，最常见的使用场景是 STL 算法、自定义排序、回调注册、线程任务封装。它的优势是代码更局部、更简洁，也能通过捕获列表访问外部上下文。

## 第三部分：内存、操作系统、平台架构

### 14. 对内存管理的理解

从 C++ 工程角度，内存管理包括：

1. **对象生命周期管理**
2. **栈内存和堆内存的使用**
3. **资源释放时机**
4. **避免泄漏、悬空、重复释放**
5. **提升局部性和性能**

#### 14.1 常见内存区域

- 栈：局部变量、函数参数，自动回收
- 堆：动态申请的对象，手动或智能指针释放
- 全局/静态区：全局变量、静态变量
- 常量区：字符串常量等
- 代码区：程序指令

#### 14.2 工程原则

- 尽量少直接写 `new/delete`
- 优先用 RAII
- 明确对象所有权
- 减少不必要拷贝
- 注意容器失效问题
- 注意多线程下共享对象生命周期

#### 14.3 面试回答模板

> 我理解内存管理不只是 `new/delete`，而是对象生命周期、资源所有权、异常安全和性能的综合问题。现代 C++ 更提倡通过 RAII 和智能指针来管理资源，减少手工内存操作带来的风险。

### 15. 虚拟内存和物理内存的异同

#### 15.1 物理内存

物理内存就是实际的 RAM 芯片空间。

#### 15.2 虚拟内存

虚拟内存是操作系统给每个进程提供的一套**独立地址空间视图**。

程序看到的是虚拟地址，CPU 通过 MMU + 页表把虚拟地址转换成物理地址。

#### 15.3 为什么要有虚拟内存

1. 进程隔离
2. 简化编程模型
3. 按页加载，提高利用率
4. 支持交换和映射文件

#### 15.4 区别总结

| 项目 | 虚拟内存 | 物理内存 |
|---|---|---|
| 是否真实存在 | 逻辑抽象 | 实际硬件 |
| 地址是否连续 | 对进程通常表现为连续 | 实际可能离散 |
| 是否隔离 | 是 | 否 |
| 管理者 | OS + MMU | 硬件 |

#### 15.5 面试回答模板

> 虚拟内存是操作系统提供给进程的逻辑地址空间，物理内存是真实 RAM。程序访问的是虚拟地址，CPU 再通过页表映射到物理地址。虚拟内存的核心价值是进程隔离、简化内存管理和支持按需分页。

这个题考的是**操作系统**和**CPU 架构**的区分。

#### 16.1 操作系统 vs 指令集架构

- **Windows / Linux**：操作系统
- **x86 / ARM**：CPU 指令集架构

不是一个维度。

#### 16.2 可以怎么组合

- Windows + x86
- Windows + ARM
- Linux + x86
- Linux + ARM

#### 16.3 `x86` 和 `ARM` 的区别

- `x86`：传统 PC / 服务器常见，历史包袱重，生态成熟
- `ARM`：低功耗优势明显，手机、嵌入式、部分服务器和 PC 越来越常见

## 第四部分：面向对象、虚函数、多态

### 17. 一道面向对象的场景题怎么答

这类题一般不会只考语法，而是考你能否把现实问题抽象成类和接口。

#### 17.1 通用答题框架

1. 先找实体
2. 抽象公共基类
3. 明确继承关系
4. 明确哪些行为适合做虚函数
5. 明确组合关系，而不是一味继承
6. 说明扩展性和开闭原则

#### 17.2 示例：图形系统

```cpp
class Shape {
public:
    virtual ~Shape() = default;
    virtual double area() const = 0;
    virtual void draw() const = 0;
};

class Circle : public Shape {
public:
    explicit Circle(double r) : r_(r) {}

    double area() const override {
        return 3.1415926 * r_ * r_;
    }

    void draw() const override {
        std::cout << "draw circle\n";
    }

private:
    double r_;
};
```

#### 17.3 面试回答模板

> 面向对象场景题我一般会先抽象实体和职责，再区分哪些是共性、哪些是差异化行为。共性抽到基类或接口里，差异化行为通过虚函数让子类实现。如果对象之间是“拥有关系”，我会优先考虑组合而不是继承。

### 18. 什么是多态，`dynamic_cast` 原理

#### 18.1 多态是什么

多态的本质：

> 同一接口，不同对象表现出不同的行为。

在 C++ 中，常说的是**运行时多态**：
- 基类指针 / 引用指向子类对象
- 调用虚函数时发生动态绑定

```cpp
class Base {
public:
    virtual void func() { std::cout << "Base\n"; }
    virtual ~Base() = default;
};

class Derived : public Base {
public:
    void func() override { std::cout << "Derived\n"; }
};
```

#### 18.2 `dynamic_cast` 是什么

`dynamic_cast` 用于**带继承关系的安全向下转型**。

```cpp
Base* p = new Derived();
Derived* d = dynamic_cast<Derived*>(p);
```

如果转换失败：
- 指针版返回 `nullptr`
- 引用版抛出 `std::bad_cast`

#### 18.3 原理

`dynamic_cast` 依赖 **RTTI（运行时类型信息）** 和 **虚函数表相关信息**。

前提：
- 基类必须是多态类型，也就是至少有一个虚函数

编译器会在对象里放 `vptr` 指向虚表，虚表关联类型信息。`dynamic_cast` 在运行时通过这些信息检查真实对象类型是否兼容。

#### 18.4 面试回答模板

> 多态就是基类接口统一，子类行为各自实现。在 C++ 里主要靠虚函数实现运行时多态。`dynamic_cast` 则是在多态体系里做安全类型转换，它依赖 RTTI 和虚表相关的运行时类型信息来判断真实对象类型。

### 19. 重载函数，返回值可以不同吗

#### 19.1 什么是重载

函数重载是指：

> 同一作用域内，函数名相同，但参数列表不同。

“参数列表不同”包括：
- 参数个数不同
- 参数类型不同
- 参数顺序不同

#### 19.2 返回值能否单独作为重载依据

**不能。**

```cpp
int func(int a);
double func(int a); // 错误
```

因为调用 `func(10)` 时，编译器无法仅根据返回值决定调用哪个函数。

#### 19.3 面试回答模板

> 函数重载要求函数名相同但参数列表不同，返回值不同本身不能构成重载条件。因为调用表达式在大多数情况下不会用返回值参与重载决议。

---

### 20. 重载和重写的区别

#### 20.1 重载（Overload）

- 发生在同一作用域
- 函数名相同
- 参数列表不同
- 与继承无关
- 属于编译期多态

#### 20.2 重写（Override）

- 发生在继承体系中
- 子类重写父类虚函数
- 函数签名要匹配
- 返回类型通常要相同或协变
- 属于运行时多态

#### 20.3 易混点：重定义 / 隐藏

如果子类定义了同名函数，但签名不匹配，可能不是重写，而是**名字隐藏**。

```cpp
class Base {
public:
    virtual void func(int) {}
};

class Derived : public Base {
public:
    void func(double) {} // 不是 override，而是隐藏
};
```

#### 20.4 面试回答模板

> 重载是同一作用域下函数名相同、参数不同，属于编译期多态；重写是子类对父类虚函数重新实现，属于运行时多态。面试里要特别注意重写必须建立在虚函数基础上，否则只是隐藏，不是 override。

### 21. 构造函数和析构函数可以是虚函数吗，同时讲虚函数原理

#### 21.1 构造函数能是虚函数吗

**不能。**

原因：
- 调用构造函数时，对象还没构造完成
- 虚函数依赖虚表指针 `vptr`
- 构造阶段对象动态类型尚未完整建立

#### 21.2 析构函数能是虚函数吗

**可以，而且基类析构函数在多态场景下通常必须是虚函数。**

```cpp
class Base {
public:
    virtual ~Base() {
        std::cout << "Base dtor\n";
    }
};

class Derived : public Base {
public:
    ~Derived() override {
        std::cout << "Derived dtor\n";
    }
};
```

如果基类析构函数不是虚函数：

```cpp
Base* p = new Derived();
delete p; // 只调用 Base 析构，可能资源泄漏
```

#### 21.3 虚函数原理

编译器通常通过：

- **vptr**：对象中的虚表指针
- **vtable**：虚函数表

来实现动态绑定。

当对象调用虚函数时：
1. 先找到对象中的 `vptr`
2. 再找到对应虚表项
3. 调用真实类型对应的函数地址

#### 21.4 面试回答模板

> 构造函数不能是虚函数，因为对象构造时虚表机制还没完全建立；析构函数可以而且在多态基类中通常应该声明成虚函数，否则通过基类指针删除子类对象时会析构不完整。虚函数底层一般通过对象里的 `vptr` 和类对应的 `vtable` 实现动态绑定。

## 第五部分：进程线程、线程池、可重入

### 22. 线程与进程的区别

#### 22.1 进程

进程是**资源分配的基本单位**。

每个进程有自己独立的：
- 地址空间
- 文件描述符表（逻辑上独立）
- 全局变量副本
- 堆、栈、代码段等

#### 22.2 线程

线程是**CPU 调度的基本单位**。

同一进程中的线程共享：
- 地址空间
- 代码段
- 堆
- 全局变量

但每个线程有自己的：
- 线程栈
- 寄存器上下文
- 程序计数器

#### 22.3 区别总结

| 项目 | 进程 | 线程 |
|---|---|---|
| 资源拥有 | 独立 | 共享所属进程资源 |
| 调度单位 | 不如线程轻量 | CPU 直接调度 |
| 切换成本 | 高 | 低 |
| 通信 | IPC 较复杂 | 共享内存更直接 |
| 稳定性 | 一个进程挂掉一般不影响别的进程 | 一个线程写坏内存可能拖垮整个进程 |

#### 22.4 面试回答模板

> 进程是资源分配的基本单位，线程是调度的基本单位。线程比进程轻量，切换更快，但线程间共享地址空间，所以同步问题更突出。进程隔离性更好，线程通信更高效。

### 23. 了解线程池吗

#### 23.1 为什么要有线程池

如果每来一个任务就创建一个线程，会有问题：

- 创建和销毁线程开销大
- 系统线程数容易失控
- 调度开销大
- 容易把 CPU 打爆

线程池的核心思想：

> 预先创建一批线程，任务来了放入队列，由工作线程重复取任务执行。

#### 23.2 线程池核心组成

1. 任务队列
2. 工作线程集合
3. 互斥锁 + 条件变量
4. 停止标记
5. 可选：future / packaged_task

#### 23.3 简化版线程池示例

```cpp
#include <vector>
#include <queue>
#include <thread>
#include <mutex>
#include <condition_variable>
#include <functional>
#include <atomic>

class ThreadPool {
public:
    explicit ThreadPool(size_t n) : stop_(false) {
        for (size_t i = 0; i < n; ++i) {
            workers_.emplace_back([this] {
                while (true) {
                    std::function<void()> task;
                    {
                        std::unique_lock<std::mutex> lock(mtx_);
                        cv_.wait(lock, [this] {
                            return stop_ || !tasks_.empty();
                        });

                        if (stop_ && tasks_.empty()) return;

                        task = std::move(tasks_.front());
                        tasks_.pop();
                    }
                    task();
                }
            });
        }
    }

    ~ThreadPool() {
        {
            std::lock_guard<std::mutex> lock(mtx_);
            stop_ = true;
        }
        cv_.notify_all();
        for (auto& t : workers_) {
            if (t.joinable()) t.join();
        }
    }

    template <typename F>
    void enqueue(F&& f) {
        {
            std::lock_guard<std::mutex> lock(mtx_);
            tasks_.emplace(std::forward<F>(f));
        }
        cv_.notify_one();
    }

private:
    std::vector<std::thread> workers_;
    std::queue<std::function<void()>> tasks_;
    std::mutex mtx_;
    std::condition_variable cv_;
    bool stop_;
};
```

#### 23.4 面试回答模板

> 线程池的目的是复用线程，避免频繁创建销毁线程的成本。通常由任务队列、工作线程、锁、条件变量和停止机制组成。线程池广泛用于服务器请求处理、异步任务执行、日志系统等。

### 24. 什么是可重入函数

这个问题常出现在嵌入式、系统、并发场景。

#### 24.1 定义

可重入函数是指：

> 一个函数在执行过程中，可以被中断后再次进入，且不会因为共享状态被破坏而产生错误。

#### 24.2 常见要求

可重入函数通常满足：
- 不使用静态/全局可变数据
- 不返回指向静态内部缓冲区的指针
- 不依赖不可重入库函数
- 调用过程只操作局部变量或调用方提供的数据

#### 24.3 不可重入示例

```cpp
char* bad_func() {
    static char buf[100];
    std::strcpy(buf, "hello");
    return buf;
}
```

问题：多个调用会共享 `buf`。

#### 24.4 可重入写法

```cpp
void good_func(char* out, size_t size) {
    std::snprintf(out, size, "hello");
}
```

#### 24.5 可重入 vs 线程安全

两者不是一回事：

- **线程安全**：多个线程同时调用不会出错
- **可重入**：函数执行中途被打断，再次进入也不会出错

例如：
- 一个函数靠加锁实现线程安全，但如果中断后再次进入同一把锁，可能死锁，不一定可重入。

#### 24.6 面试回答模板

> 可重入函数强调函数可以在尚未执行完时被再次进入而不出错，通常要求不依赖共享可变状态。它和线程安全不完全等价，线程安全可以通过加锁实现，但加锁函数未必可重入。

### 26. 如何在 GDB 中查看崩溃点变量状态

### 27. git 如何回退到之前的 commit

#### 27.1 几种常见方式

##### 方式 1：`git reset --soft`

回退 commit，但保留修改到暂存区：

```bash
git reset --soft HEAD~1
```

##### 方式 2：`git reset --mixed`

回退 commit，保留修改到工作区：

```bash
git reset --mixed HEAD~1
```

##### 方式 3：`git reset --hard`

彻底回退，工作区也回去：

```bash
git reset --hard HEAD~1
```

##### 方式 4：回退到指定 commit

```bash
git reset --hard <commit_id>
```

##### 方式 5：如果已经推到远端，优先 `git revert`

```bash
git revert <commit_id>
```

这会生成一个“反向提交”，更安全。

#### 27.2 面试回答模板

> 本地回退常用 `git reset`，其中 `--soft` 保留暂存区，`--mixed` 保留工作区，`--hard` 会直接丢弃修改。如果 commit 已经推到共享分支，我一般更倾向于 `git revert`，因为它不会重写历史。

## 第七部分：STL 容器与算法基础

### 30. vector 与 list 的区别

| 对比项 | vector | list |
|---|---|---|
| 底层结构 | 动态数组 | 双向链表 |
| 内存 | 连续 | 非连续 |
| 随机访问 | 支持，O(1) | 不支持 |
| 尾插 | 均摊 O(1) | O(1) |
| 中间插入删除 | O(n) | 已知位置 O(1) |
| 缓存友好 | 好 | 差 |
| 额外空间 | 少 | 指针开销大 |

#### 面试回答模板

> `vector` 底层是连续数组，支持随机访问，缓存友好，绝大部分场景下是首选；`list` 底层是链表，适合频繁插入删除且已经定位到节点位置的场景，但不支持随机访问，实际性能也常常不如 `vector`。

### 31. map 与 unordered_map 的区别

#### 31.1 `map`

- 底层一般是**红黑树**
- 有序
- 查找 / 插入 / 删除复杂度 `O(log n)`

#### 31.2 `unordered_map`

- 底层一般是**哈希表**
- 无序
- 平均查找 / 插入 / 删除 `O(1)`
- 最坏退化到 `O(n)`

#### 31.3 对比

| 对比项 | map | unordered_map |
|---|---|---|
| 底层 | 红黑树 | 哈希表 |
| 是否有序 | 有序 | 无序 |
| 查找复杂度 | O(log n) | 平均 O(1) |
| 是否支持范围查询 | 支持 | 不适合 |
| 内存占用 | 相对稳定 | 桶 + 节点，可能更高 |

#### 31.4 面试回答模板

> `map` 是基于红黑树的有序关联容器，适合有序遍历、范围查找；`unordered_map` 是哈希表，平均查找更快，但无序，且哈希冲突严重时可能退化。

### 32. vector 中 `resize` 与 `reserve` 的区别

你题目里写的是 `resize` 和 `reverse`，但面试里**高概率真实想问的是 `resize` 和 `reserve`**。这里两个都讲。

#### 32.1 `resize`

- 改变的是 **容器中元素个数 size**
- 如果变大，会新增元素（默认构造或填充值）
- 如果变小，会销毁多余元素

```cpp
std::vector<int> v{1, 2, 3};
v.resize(5); // 变成 {1,2,3,0,0}
```

#### 32.2 `reserve`

- 改变的是 **容量 capacity**
- 不改变现有元素个数 `size`
- 用于提前分配空间，减少扩容次数

```cpp
std::vector<int> v;
v.reserve(100);
```

#### 32.3 `reverse`

`reverse` 不是 `vector` 成员函数，而是算法：

```cpp
std::reverse(v.begin(), v.end());
```

它的作用是**反转区间内元素顺序**。

#### 32.4 面试回答模板

> `resize` 改的是元素个数，可能会构造或销毁元素；`reserve` 改的是容量，不改变 size，常用来提前预留空间避免多次扩容。`reverse` 则是算法，不是容量管理接口，它只是把元素顺序反转。

### 34. 手撕冒泡排序

#### 34.1 基本思想

每一轮比较相邻两个元素，把较大的“冒”到后面。

#### 34.2 代码实现

```cpp
#include <vector>
#include <iostream>

void bubbleSort(std::vector<int>& nums) {
    int n = static_cast<int>(nums.size());
    for (int i = 0; i < n - 1; ++i) {
        bool swapped = false;
        for (int j = 0; j < n - 1 - i; ++j) {
            if (nums[j] > nums[j + 1]) {
                std::swap(nums[j], nums[j + 1]);
                swapped = true;
            }
        }
        if (!swapped) break;
    }
}

int main() {
    std::vector<int> nums{5, 1, 4, 2, 8};
    bubbleSort(nums);
    for (int x : nums) {
        std::cout << x << " ";
    }
}
```

#### 34.3 复杂度

- 时间复杂度：`O(n^2)`
- 空间复杂度：`O(1)`
- 稳定排序

#### 34.4 面试回答模板

> 冒泡排序通过不断交换相邻逆序元素，让最大值逐轮移动到末尾。它实现简单、稳定，但时间复杂度是 `O(n^2)`，工程里一般不会直接使用，更多是考察基础功底。

## 第八部分：手撕 String 类（Rule of Five）

### 35. 手撕 `string` 类：实现拷贝构造 / 拷贝赋值 / 移动构造 / 移动赋值 / 析构

这个题本质是在考：

- 深拷贝
- 资源所有权
- Rule of Five
- 异常安全意识

#### 35.1 代码示例

```cpp
#include <cstring>
#include <iostream>
#include <utility>

class String {
public:
    String() : data_(new char[1]{'\0'}), size_(0) {}

    explicit String(const char* s) {
        if (!s) {
            data_ = new char[1]{'\0'};
            size_ = 0;
        } else {
            size_ = std::strlen(s);
            data_ = new char[size_ + 1];
            std::memcpy(data_, s, size_ + 1);
        }
    }

    // 拷贝构造
    String(const String& other) : data_(new char[other.size_ + 1]), size_(other.size_) {
        std::memcpy(data_, other.data_, size_ + 1);
    }

    // 移动构造
    String(String&& other) noexcept : data_(other.data_), size_(other.size_) {
        other.data_ = new char[1]{'\0'};
        other.size_ = 0;
    }

    // 拷贝赋值：copy-and-swap 写法更稳
    String& operator=(const String& other) {
        if (this == &other) return *this;
        String temp(other);
        swap(temp);
        return *this;
    }

    // 移动赋值
    String& operator=(String&& other) noexcept {
        if (this == &other) return *this;
        delete[] data_;
        data_ = other.data_;
        size_ = other.size_;
        other.data_ = new char[1]{'\0'};
        other.size_ = 0;
        return *this;
    }

    ~String() {
        delete[] data_;
    }

    const char* c_str() const {
        return data_;
    }

    size_t size() const {
        return size_;
    }

    void swap(String& other) noexcept {
        std::swap(data_, other.data_);
        std::swap(size_, other.size_);
    }

private:
    char* data_;
    size_t size_;
};

int main() {
    String s1("hello");
    String s2 = s1;             // 拷贝构造
    String s3 = std::move(s1);  // 移动构造

    String s4;
    s4 = s2;                    // 拷贝赋值

    String s5;
    s5 = std::move(s3);         // 移动赋值

    std::cout << s2.c_str() << std::endl;
    std::cout << s4.c_str() << std::endl;
    std::cout << s5.c_str() << std::endl;
}
```

#### 35.2 面试讲解点

- 为什么要深拷贝：避免多个对象共用同一块堆内存，析构时重复释放
- 为什么移动构造要 `noexcept`：标准容器扩容时更愿意用 noexcept 移动
- 为什么拷贝赋值用 copy-and-swap：异常安全更好
- 为什么要处理自赋值：`a = a`

#### 35.3 面试回答模板

> 如果手写一个 `String` 类，我会重点处理五个特殊成员函数，也就是 Rule of Five：析构、拷贝构造、拷贝赋值、移动构造、移动赋值。因为这个类自己管理堆内存，必须保证深拷贝、避免重复释放，并通过移动语义减少不必要拷贝。

## 第九部分：网络 IO

### 36. select / poll / epoll 的特点和异同

#### 36.1 `select`

特点：
- 最早、最通用
- 通过位图表示 fd 集合
- 每次调用都要把用户态集合拷到内核态
- 返回后还要遍历所有 fd
- 有最大 fd 数量限制（通常 1024）

#### 36.2 `poll`

特点：
- 用数组描述 fd，突破了 `select` 的位图限制
- 依然每次都要拷贝数据结构
- 依然需要线性遍历所有 fd

#### 36.3 `epoll`

特点：
- Linux 特有
- 事件注册和等待分离
- 内核维护就绪事件
- 不需要每次传整个 fd 集合
- 就绪时只返回活跃 fd
- 适合高并发场景

#### 36.4 LT 和 ET

`epoll` 有两种模式：

- LT（Level Trigger）：默认，水平触发
- ET（Edge Trigger）：边沿触发，效率更高，但要求一次性读干净

#### 36.5 对比表

| 项目 | select | poll | epoll |
|---|---|---|---|
| 平台 | 跨平台 | 跨平台 | Linux |
| fd 数量限制 | 有 | 无明显固定上限 | 无明显固定上限 |
| 每次调用拷贝集合 | 是 | 是 | 否 |
| 是否遍历全部 fd | 是 | 是 | 否，只返回就绪 fd |
| 复杂度 | O(n) | O(n) | 平均更优 |
| 适用场景 | 小规模兼容场景 | 中等规模 | 大规模高并发 |

#### 36.6 面试回答模板

> `select`、`poll`、`epoll` 都是 IO 多路复用机制。`select` 和 `poll` 本质上都需要用户态把监听集合传给内核并线性遍历，复杂度接近 `O(n)`；`epoll` 则把事件注册和事件等待分离，内核只返回就绪 fd，更适合高并发服务器场景。

## 附录：一句话速背版

- 队列：FIFO，头出尾进，常用于任务调度和生产者消费者模型。  
- `vector`：连续内存，随机访问快，扩容可能搬迁。  
- 链表：插删方便，但不支持随机访问，缓存不友好。  
- 线程安全队列：锁 + 条件变量。  
- RAII：资源绑定对象生命周期。  
- `unique_ptr`：独占所有权，只能移动。  
- `shared_ptr`：共享所有权，引用计数。  
- `weak_ptr`：弱引用，用来打破循环引用。  
- 左值右值：左值可持久存在，右值通常是临时对象。  
- 移动语义：转移资源，避免深拷贝。  
- `std::forward`：保留值类别，实现完美转发。  
- 重载：同名不同参；重写：子类改写父类虚函数。  
- 构造函数不能是虚函数；析构函数在多态基类中应为虚函数。  
- 多态：基类接口统一，子类行为不同。  
- `dynamic_cast`：依赖 RTTI 的安全向下转型。  
- 进程是资源分配单位，线程是调度单位。  
- 可重入函数：可被中断后再次进入也安全。  
- 线程池：复用线程，减少创建销毁开销。  
- `select/poll/epoll`：IO 多路复用，`epoll` 更适合高并发。  
- `map` 有序红黑树；`unordered_map` 无序哈希表。  
- `resize` 改 size；`reserve` 改 capacity。
