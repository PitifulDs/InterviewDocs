Agent，可以理解成“**会自己感知、思考、决策、执行的一类智能体程序**”。

如果普通大模型像“你问一句，我答一句”的聊天助手，那 Agent 更像“**拿着目标去做事的助手**”：

- 你给它一个目标：比如“帮我整理简历并投递岗位”
- 它会自己拆任务
- 决定先做什么、后做什么
- 需要时调用工具：搜索、读文档、写代码、发邮件、查日历
- 根据执行结果继续调整下一步
- 最终完成目标，而不是只停留在“回答问题”

------

# 一、什么是 Agent

更正式一点说：

**Agent = LLM（大模型） + 记忆/状态 + 规划能力 + 工具调用能力 + 执行闭环**

它通常有这几个核心特征：

### 1）有目标

不是单纯对话，而是围绕一个目标持续行动。
 比如：

- “帮我分析这个代码仓库”
- “帮我做一份面试准备计划”
- “自动监控某个网页更新”
- “根据用户问题查数据库并生成报告”

### 2）会感知环境

它能读取外部信息：

- 用户输入
- 文件
- 数据库
- API 返回结果
- 网页内容
- 系统状态

### 3）会做决策

Agent 不只是生成一段文字，而是会判断：

- 现在缺什么信息
- 下一步该调用哪个工具
- 当前结果够不够
- 是否需要重试、修正路径

### 4）会执行动作

比如：

- 调搜索工具
- 调代码执行器
- 查企业知识库
- 写文件
- 发请求
- 调用业务接口

### 5）有反馈闭环

执行完以后，它会看结果：

- 成功了吗
- 输出是否满足目标
- 需不需要继续下一轮

这就是 Agent 和普通 Prompt 的核心区别。

------

# 二、一个最简单的理解

你可以把 Agent 看成一个“AI版程序员助理”：

用户说：
 “帮我看看这个项目为什么训练报错，并给出修复建议。”

普通聊天模型通常会：

- 读你贴的报错
- 给出一些推测

而 Agent 可能会：

- 先读取错误日志
- 再打开相关代码文件
- 分析调用链
- 查配置文件
- 运行测试脚本
- 对比输入输出
- 最后给出修复方案，甚至自动改代码

所以：

**Prompt 更像问答**
 **Agent 更像做任务**

# 三、Agent 的典型组成

一个 Agent 常见架构可以拆成下面几层：

## 1）模型层

核心大脑，通常是 LLM。

它负责：

- 理解任务
- 推理
- 生成计划
- 决定调用什么工具
- 总结结果

常见模型：

- GPT 系列
- Claude
- Qwen
- Llama
- DeepSeek 等

------

## 2）记忆层

Agent 往往不只是看当前一句话，还要保留上下文和历史。

常见记忆分成：

### 短期记忆

当前会话上下文、最近操作结果。

比如：

- 上一步搜到了什么
- 当前任务执行到哪一步
- 用户刚刚补充了哪些约束

### 长期记忆

持久化知识或用户偏好。

比如：

- 用户偏好 C++
- 用户最近在做 EdgeLLM 项目
- 某个项目的历史分析结果

实现方式可能是：

- 直接存在数据库
- 向量数据库
- KV 存储
- 文件系统

------

## 3）规划层

负责把大目标拆成小步骤。

例如用户说：

“帮我找自动驾驶 C++ 岗位，并按匹配度排序。”

规划层可能拆成：

1. 提取岗位关键词
2. 搜索职位
3. 提取岗位要求
4. 和用户简历做匹配
5. 生成排序结果
6. 输出建议

更复杂的 Agent 还会做：

- 任务树分解
- 多步推理
- 自我反思
- 路径重规划

------

## 4）工具层

这是 Agent 很关键的一层。

因为模型本身只会“生成文本”，不会天然拥有真实世界操作能力。
 所以要给它接工具。

常见工具：

- Web Search：联网搜索
- File Reader：读 PDF / Word / 代码
- Python Executor：运行代码
- Shell Tool：执行命令
- Database Query：查数据库
- Email / Calendar：邮件、日程
- Browser / RPA：网页操作
- Internal API：企业内部接口

本质上，工具就是：

**把模型的“想法”变成“系统可执行动作”**

------

## 5）执行控制层

负责协调整个循环：

1. 接收目标
2. 调模型分析
3. 调工具执行
4. 获取结果
5. 再调模型判断下一步
6. 直到完成或中止

这一层一般由程序逻辑控制，不是纯模型完成。

------

# 四、Agent 和工作流（Workflow）的区别

很多人容易把 Agent 和 Workflow 混在一起。

其实区别很重要：

## Workflow

步骤是提前写死的。

例如：

1. 用户上传简历
2. 提取文本
3. 匹配岗位 JD
4. 生成评分
5. 输出报告

这是固定流程，可预测、稳定。

## Agent

步骤不一定提前写死，会动态决定。

比如：

- 如果 JD 信息不够，就先上网查公司
- 如果简历缺项目描述，就先提示补充
- 如果评分依据不明确，就先拆能力维度

所以：

**Workflow 更偏工程自动化**
 **Agent 更偏自主决策**

实际项目里，最常见的是：

**Workflow + Agent 混合**

也就是：

- 大流程是固定的
- 某些关键节点让 Agent 自主决策

这是最实用的工程方案。

# 五、做一个 Agent 需要哪些基础知识

这个问题很关键。
 我建议分成 4 层看：**编程基础、AI基础、系统基础、工程基础**。

------

## 第一层：编程基础

这是最基础的。

### 1）至少熟悉一门主语言

最常见是：

- Python：Agent 生态最成熟
- JavaScript / TypeScript：做前后端 Agent 应用也很多
- C++：如果你要做高性能推理框架、边缘部署、执行引擎，也很重要

如果你是想“快速做 Agent 原型”，优先建议 **Python**。

你至少要会：

- 函数、类、模块
- 异步/并发基础
- HTTP 请求
- JSON 处理
- 文件读写
- 异常处理
- 日志打印

------

### 2）数据结构与算法基础

不用刷到竞赛级，但这些要懂：

- 数组、哈希表、队列、栈
- 树、图的基本思想
- 搜索、排序
- 状态机
- 任务调度基础

因为 Agent 常常会涉及：

- 任务队列
- 工具路由
- 状态管理
- 多轮执行控制

------

## 第二层：AI / LLM 基础

### 1）知道大模型到底能做什么、不能做什么

你要理解：

- LLM 擅长语言理解、推理、总结、规划
- 但容易幻觉
- 没有天然外部世界权限
- 不一定可靠执行精确逻辑
- 长上下文也不是无限可靠

这决定了 Agent 设计方式。

------

### 2）Prompt Engineering 基础

至少要会：

- 角色设定
- 任务约束
- 输出格式约束
- Few-shot 示例
- 思维链/分步要求
- 工具调用格式设计

因为 Agent 的“决策质量”，很大程度受 Prompt 影响。

------

### 3）Function Calling / Tool Calling

这几乎是 Agent 开发必备。

你要懂：

- 如何定义工具 schema
- 模型怎么选择工具
- 工具参数怎么传
- 工具结果怎么回注给模型
- 如何处理工具失败/重试

这是从“会聊天”走向“会做事”的关键一步。

------

### 4）RAG 基础

RAG = Retrieval Augmented Generation，检索增强生成。

很多 Agent 并不是纯靠模型记忆，而是：

- 先查资料
- 再基于资料回答/决策

你要懂：

- 文本切分
- 向量化 embedding
- 向量检索
- rerank
- chunk 召回
- 上下文拼接

如果 Agent 需要读文档、查知识库，这块非常重要。

------

### 5）多轮推理与规划

你要理解一些常见范式：

- ReAct：Reason + Act
- Plan-and-Execute
- Toolformer 思路
- Reflection / Self-correction
- Multi-agent 协作

不用上来就把论文全啃完，但这些概念要知道。

------

## 第三层：系统与工程基础

### 1）API 与服务开发

大多数 Agent 最后都不是一个脚本，而是一个服务。

你要会：

- REST API
- WebSocket / SSE
- 请求响应模型
- 鉴权
- 超时与重试
- 服务部署

常见框架：

- FastAPI
- Flask
- Node.js / Express / NestJS

------

### 2）数据库基础

至少懂：

- MySQL / PostgreSQL
- Redis
- 向量数据库（Milvus、FAISS、Chroma、Weaviate 等）

因为 Agent 常要存：

- 会话状态
- 用户数据
- 工具结果缓存
- 记忆
- 检索索引

------

### 3）异步与并发

很重要，尤其是多工具、多任务 Agent。

你要理解：

- 同步 vs 异步
- 线程/进程基本概念
- 任务队列
- Future / Promise
- 超时控制
- 并发安全

如果不懂这块，Agent 一多轮、多工具就容易乱。

------

### 4）日志、监控、调试

Agent 最大的问题之一就是：**不好 debug**。

所以你要有工程意识：

- 每一步决策日志
- 工具调用日志
- Token 消耗
- 延迟统计
- 失败原因记录
- 中间状态可视化

否则出了问题你根本不知道是：

- Prompt 有问题
- 工具参数错了
- 检索没召回
- 模型判断失误
- 上下文污染了

------

## 第四层：Agent 专项能力

### 1）状态管理

Agent 不是一次函数调用，它是“持续执行过程”。

你要管理：

- 当前任务阶段
- 已完成步骤
- 未完成步骤
- 工具返回结果
- 重试次数
- 中止条件

本质上是在写一个“智能状态机”。

------

### 2）任务拆解能力

这部分有时由模型做，有时由程序辅助做。

比如把：

“帮我准备自动驾驶 C++ 面试”

拆成：

- 岗位方向分析
- 八股文整理
- 项目介绍优化
- 高频问答生成
- 模拟面试

------

### 3）工具设计能力

工具不是越多越好，而是要：

- 职责清晰
- 输入输出稳定
- 错误信息明确
- 粒度合适

比如不要做一个“万能工具”，而要拆成：

- `search_jobs`
- `read_resume`
- `score_match`
- `generate_report`

这样模型更容易选对。

------

### 4）失败恢复机制

真实 Agent 一定会失败。

所以你要考虑：

- 工具报错怎么办
- 查询无结果怎么办
- 模型输出不合法怎么办
- 步骤卡住怎么办
- 是否自动重试
- 是否降级到人工确认

这是工程上非常关键的点。

------

# 六、如果你想做 Agent，推荐学习路线

我给你一个很实用的路线。

## 第 1 阶段：先会调用大模型 API

目标：

- 能写一个程序调用模型

你会的东西：

- Python 基础

------

## 第 2 阶段：学会工具调用

目标：

- 让模型决定何时调用工具

练习项目：

- 做一个“天气查询助手”

------

## 第 3 阶段：学 RAG

目标：

- 能让 Agent 读本地文档或知识库

练习项目：

- PDF 问答

- 面试知识库助手

## 第 4 阶段：学多步任务执行

目标：

- 让 Agent 不只是答一句，而是能连续干活

## 第 5 阶段：工程化

- 把 Agent 做成可部署系统

- 监控
- 用户权限控制

# 七、一个最小 Agent 需要哪些模块

你可以把最小版本理解成下面这样：

```
用户输入
   ↓
任务解析
   ↓
模型决定：直接回答 or 调工具
   ↓
工具执行
   ↓
拿到结果
   ↓
模型总结 / 继续下一步
   ↓
输出最终结果
```

最小代码层面通常需要：

- 一个 LLM client
- 一个 tool registry
- 一个 agent loop
- 一个 message/state manager
- 一个 logger

# 八、初学者最容易踩的坑

### 1）一上来就做“全自动超级 Agent”

结果往往会很乱。

更好的做法是：

- 先做单任务
- 再做两三个工具
- 再加记忆和规划

------

### 2）过度相信模型

模型会一本正经地胡说。

所以一定要：

- 能验证就验证
- 工具结果尽量结构化
- 关键步骤做约束和校验

------

### 3）工具设计太模糊

比如一个工具叫：

```
do_everything()
```

模型很难稳定调用。

要把工具定义清楚。

------

### 4）没有状态管理

执行几轮后上下文乱掉，Agent 开始漂移。

所以一定要管理：

- 当前目标
- 已完成步骤
- 当前证据
- 下一步候选动作

------

### 5）没有日志

Agent 出问题时，最怕“黑盒”。

一定要把每轮：

- 输入
- 模型输出
- 工具调用
- 工具结果
- 错误

记录下来。

------

### 4）本地工具型 Agent

# 十一、一句话总结

**Agent 不是“更会聊天的模型”，而是“能围绕目标自主调用工具并完成任务的智能系统”。**

做 Agent 需要的基础知识，本质上是这四块：

- **编程基础**：Python / API / 数据结构
- **LLM 基础**：Prompt、工具调用、RAG、多轮推理
- **系统基础**：服务、数据库、异步、日志监控
- **工程能力**：状态管理、任务拆解、失败恢复、可观测性
